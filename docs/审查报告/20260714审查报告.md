### 5.1 M1 共享父 Issue

建议包含：

- 共同基线：当前核验为 `e9a0e77`；正式创建分支时再次读取，不硬编码旧值。
- 入口条件：M0 `complete`，M1 `ready`。

- 共同任务映射：T004～T020。
- 共同出口门禁：Roadmap M1 第 7 节及 Tasks 的 Setup/Foundational checkpoint。
- 非目标：业务功能、生产部署、对象存储、照片/OCR、PDF、未来子系统。
- 两个实现子 Issue 的链接和独立状态。

### 5.2 Codex / Trae 实现子 Issue

两个子 Issue 使用相同的共同范围和验收门禁，但分别记录：

- 所属分支、实际分支 HEAD、共同基线提交。
- T004～T020 的本实现执行清单与依赖顺序。
- RED 测试 seam、最小实现和验证命令。
- 各自提交、迁移/锁文件、测试结果、风险和阻塞。
- 只读交叉评审入口；不复制或 cherry-pick 对方实现。

### 5.3 不应做的拆分

- 不把 T004～T020 每项机械创建为独立 Issue。
- 不按“全部 contracts / 全部数据库 / 全部页面”建立长期水平 Issue。
- 不预先创建尚为 `pending` 的 M2～M8 实现 Issue；可保留 Roadmap 导航，但应等待前一里程碑完成并进入 `ready`。
- 不在 Issue 中宣称未执行的 `uv`、Ruff、Pyright、Pytest、Docker 或运行入口检查已通过。
