## 1. 目标与完成定义

本方案的目标不是再次修改已经收敛的产品需求，而是消除 M1 启动流程中的操作歧义：

1. 将 M1 Roadmap 范围明确映射到 T003 与 T004～T020。
2. 消除启动检查清单中“创建首个 Issue 前，Issue 已建立”的循环措辞。
3. 在执行 T003 前安全处理现有 graphify 工作区修改。
4. 在新 Issue 中写入 2026-07-14 的实时基线证据，不为日期单独制造无价值文档提交。
5. 通过引用现有双 Agent 协议统一协作边界，不复制第二套规则。

只有以下条件全部满足，才算本方案完成：

- M1 一个共享父 Issue、Codex 与 Trae 两个实现子 Issue 的内容经过只读复核。
- Issue 文案明确 T003、T004～T020、入口条件、出口门禁、非目标和独立授权边界。
- 双 Agent 启动清单不再存在字面循环。
- 现有 graphify 修改的归属和处置获得用户确认，未覆盖或丢弃用户工作。
- 正式执行 T003 时重新读取并记录两个分支共同基线提交。
- Issue、分支和编码分别获得明确授权；任何一步都不从前一步自动推导授权。

## 3. 阶段 A：形成 M1 Issue 草稿

### A1. 共享父 Issue

建议标题：`M1：工程骨架与质量基线（共享父 Issue）`

必须包含：

- 共同基线：草稿阶段写“正式执行 T003 时读取”；不得把当前短提交 ID永久硬编码为未来事实。
- 入口条件：M0 `complete`，M1 `ready`。
- 共同目标：Python 3.14 工程、依赖锁、三个运行入口、质量工具、开发 PostgreSQL/Redis、健康检查、请求/追踪 ID、依赖方向和测试替身。
- 任务映射：T003 是共享授权与分支门禁；T004～T020 是两个实现分别完成的 M1 清单。
- 出口门禁：Roadmap M1 第 7 节、Tasks Setup checkpoint 和 Foundational checkpoint。
- 非目标：业务功能、生产部署、对象存储、照片/OCR、PDF、未来子系统。
- 两个实现子 Issue 链接和独立状态。
- 双 Agent 规则引用：只读交叉评审、共享文档同步、冲突冻结均链接到现有协议，不复制第二套规范。

### A2. Codex 实现子 Issue

必须包含：

- 目标分支 `codex` 与正式创建时记录的共同基线提交。
- T004～T020 的依赖有序清单。
- RED seam、最小实现、相关快速测试、标准五条命令和 graphify 更新要求。
- Codex 自己的提交、风险、阻塞和验收证据字段。
- 禁止读取对方结果后省略自身验证，禁止复制或 cherry-pick Trae 实现。

### A3. Trae 实现子 Issue

结构与共同验收门禁必须和 Codex 子 Issue 对称，但只记录 Trae 分支的方案、提交、验证、风险和阻塞；不得共享实现提交或把 Codex 证据当作自身证据。

### A4. 草稿验收

- 三份草稿全部引用同一 Roadmap、Spec、Plan、Tasks、ADR 和模板基线。
- 父 Issue 不包含任一实现专属类名、迁移细节或提交列表。
- 两个子 Issue 的共同门禁一致，内部实现允许不同。
- T004～T020 不拆成 17 个独立 GitHub Issue。
- M2～M8 保持 `pending`，不预建实现 Issue。
- 草稿完成不改变 GitHub、分支或代码状态。

## 5. 阶段 C：处理 graphify 工作区修改

本阶段必须在 T003 前执行，并由用户选择处置方式；本方案不预设这些修改属于谁，也不自动提交或丢弃。

### C1. 建立证据

记录：

```bash
git status --short --branch
git diff --stat -- graphify-out
git diff --check -- graphify-out
```

同时记录工作区图谱节点/边数及完整性诊断，区分 HEAD 图谱与工作区图谱。

### C2. 用户选择处置

可选的非破坏性方向：

1. 经审阅后作为独立 graphify 生成物提交保存。
2. 若属于尚未完成的图谱更新，先完成验证再决定提交。
3. 使用用户指定的其他保留方式。

不得未经授权使用 `git reset --hard`、强制覆盖、删除或还原这些修改。仅创建 Issue 草稿不要求工作区先清洁；创建/切换实现分支前必须完成处置确认。

### C3. 图谱验收

- 已删除的 `codex20260714审查报告.md`、`trae20260714审查报告.md` 不再作为活动 source node 或 manifest 路径。
- `20260714审查报告.md` 与唯一收敛方案 `20260714解决方案.md` 可以定位。
- 已删除的 `codex20260714解决方案.md`、`trae20260714解决方案.md` 不再作为活动 source node 或 manifest 路径。
- M0-G1～G8、FR-031 快照原因码、Q13 幂等定义、模板反思规则及 T002/T003 双 Agent 流程等关键概念均可定位。
- 无缺失端点、悬空边、自环、异常重复边或未解释收缩。
- 图谱刷新仅作为导航验证，不替代当前 Markdown 事实来源。

## 6. 阶段 D：经授权创建 Issue

执行前：

- 获得明确的 GitHub Issue 操作授权。
- 实时查询全部 Issue，确认没有并行创建的重复 M1 条目。
- 复核父子 Issue 标题、正文、依赖、验收、非目标和授权声明。

执行顺序：

1. 创建 M1 共享父 Issue。
2. 创建并关联 Codex、Trae 两个实现子 Issue；二者无业务先后顺序。
3. 回填父 Issue 中的两个子 Issue 链接。
4. 只记录 `ready`/待授权状态，不把 Issue 创建写成实现已开始。

验收：三个 Issue 均可访问、层级正确、无重复；任何 Issue 操作结果如实记录。完成本阶段不自动授权 T003。

## 7. 阶段 E：经授权执行 T003

执行前必须同时满足：

- 分支操作已获得明确授权。
- graphify 现有修改已经按阶段 C 确认处置。
- `main` 与 `origin/main` 状态已重新读取并符合用户确认的共同基线。
- M1 三个 Issue 已建立或用户明确指定先建分支后建 Issue。

执行要求：

- `codex` 与 `trae` 从同一实际 `main` 提交创建。
- 两个子 Issue 记录完全相同的共同基线提交 ID。
- 初始建分支不得 cherry-pick 旧基线。
- 两个实现分支不得互相合并、cherry-pick 或复制实现。
- 不在 `main` 开始 T004 或业务实现。

完成 T003 不自动授权开始编码。

## 8. 阶段 F：经授权开始 M1 实现

两个 Agent 分别在自己的实现分支执行 T004～T020：

1. 严格按测试驱动开发（TDD）先建立可观察的 RED seam，确认失败由当前任务缺失行为导致。
2. 只实现使当前任务转为 GREEN 的最小行为，不提前实现未来里程碑或无业务依赖的抽象。
3. 运行相关快速测试。
4. 在 checkpoint 运行标准五条命令及对应专项检查。
5. 更新 graphify 并记录失败原因或完整性结果。
6. 只向自己的子 Issue 写入提交、验证、风险和阻塞。

任何共享规格冲突按双 Agent 协议第 7 节暂停受影响工作，先回到 `main` 确认文档，不允许两个实现分支私下统一需求。

## 10. 停止条件

出现以下任一情况必须暂停并报告：

- canonical 文档之间出现新的事实冲突。
- Issue 草稿需要扩大到 M2～M8 或生产部署范围。
- GitHub 已存在重复或并行创建的 M1 Issue。
- graphify 修改归属不明，或更新出现未解释收缩/损坏。
- `main` 与 `origin/main` 不一致，无法确定共同基线。
- 任一操作需要尚未获得的 Issue、分支、提交、推送或实现授权。
- 发现新的密钥、个人信息或敏感正文进入当前文件或历史。

## 11. 推荐执行顺序

```text
三份 Issue 草稿与只读复核
  → 澄清启动清单
  → 确认 graphify 现有修改归属并完成验证
  → 单独授权创建 M1 父子 Issue
  → 单独授权执行 T003
  → 两个子 Issue 记录共同基线
  → 单独授权开始 T004～T020
```

该顺序保持 Issue、分支和编码授权相互独立，同时保证两个实现从同一已确认共享基线开始。
