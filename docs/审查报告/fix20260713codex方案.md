# M0 阻塞项修复执行方案

> 制定日期：2026-07-13  
> 依据文件：[编码前联合审查报告](codex-trae联合审查报告20260713.md)
> 操作边界：本方案只定义执行步骤，不授权修改项目事实来源、提交、推送、创建或切换分支、创建 Issue 或改写 Git 历史。

本方案按“内容修复 → 状态收敛 → 专项验证 → 历史清理 → 最终基线 → M1 启动”的依赖顺序执行。不得先修改完成状态或关闭门禁，否则会出现文档声称完成、验证证据仍未通过的情况。

## 1. 阶段 0：修复前基线确认

目标：避免覆盖现有工作区修改，并为模板和文档变化建立对照证据。

执行：

```bash
git status --short --branch
git diff --check
sha256sum templates/teacherplan/teacherplan.docx
```

记录：

- 当前分支和 HEAD。
- 当前工作区已有修改。
- 模板 SHA-256。
- 当前 README 失效链接。
- 当前 graphify 节点、边和诊断结果。

完成标准：确认本轮只修改授权范围内的 Markdown，不修改 `.docx` 模板。

## 2. 阶段 1：任务 A——修复内容缺口

四项普通文档修复在同一阶段完成。

### 2.1 A1：修订模板说明

目标文件：`templates/teacherplan/一日活动计划系统说明.md`

修改：

- 更新固定 HTTPS 节假日地址。
- 删除 `[[...]]` 标记。
- 集体活动使用 `is_ai_added=true`。
- 正文不得写入“新增”等标记文字。
- 反思使用 `highlights`、`issues`、`adjustments`。
- 外部 AI 输入遵循数据最小化原则，只发送五个必要上游栏目和最小必要上下文。
- 加入 Unicode NFKC、完整 Unicode 码点和合计 200 上限。
- 删除 BMP 旧规则。
- 明确 Word 三行映射。

快速验收：

```bash
rg -n "http://timor.tech|\\[\\[|U\\+0000|U\\+FFFF|截断|替换" templates/teacherplan/一日活动计划系统说明.md
rg -n "https://timor.tech/api/holiday/info|is_ai_added|highlights|issues|adjustments|NFKC|200|emoji" templates/teacherplan/一日活动计划系统说明.md
```

第一个检索不应再有旧规则命中；第二个检索必须找到全部新规则。

### 2.2 A2：补充 FR-031

目标文件：`specs/001-daily-activity-plan/spec.md`

修改：

- 添加 `manual_save`、`ai_adopted`、`archive`、`unarchive`、`before_restore`、`restored` 六个原因码。
- 明确 `before_restore` 发生在覆盖当前内容前。
- 明确 `restored` 发生在新当前内容形成后。
- 只有采用仍有效的 AI 预览时使用 `ai_adopted`。
- 自动保存和 AI 生成过程不创建快照。
- 与 FR-032 保持一致。

验收：Spec、数据模型和数据库 Schema 的原因码及恢复顺序完全一致。

### 2.3 A3：补充 Q13 架构摘要

目标文件：`docs/design/system-architecture.md`

修改：

- 概括客户端幂等 scope。
- 概括 path、query、body fingerprint。
- 明确相同请求重放与 fingerprint 冲突行为。
- 明确内部子任务不伪造客户端幂等键。
- 引用 OpenAPI 和契约说明，不复制第二套机器契约。

验收：架构摘要能够导航到：

```text
specs/001-daily-activity-plan/contracts/openapi.yaml
specs/001-daily-activity-plan/contracts/README.md
```

### 2.4 A4：修复 README 链接

目标文件：`README.md`

将两个根目录 `contracts/` 链接改为 Spec Kit 契约路径。

阶段出口：

- 内容缺口已修复。
- 模板原件哈希不变。
- 尚不修改 M0 为完成状态。

## 4. 阶段 3：任务 C——执行 P1 专项验证

### 4.1 C1：静态文档和模板检查

执行：

```bash
git diff --check
sha256sum templates/teacherplan/teacherplan.docx
```

另外验证：

- Markdown 缺失链接为 0。
- 模板 SHA-256 与全部固定引用一致。
- `.docx` 没有工作区变化。
- 模板表格结构、字体、字号、段落和换行不变。
- 旧 HTTP、`[[...]]` 和 BMP 规则无残留。
- 新字段、HTTPS、NFKC、200 限制均存在。

### 4.2 C2：Spec Kit 一致性检查

执行 `speckit-analyze`，并复核：

- 72 个 FR 编号连续。
- 17 个 SC 编号连续。
- T001～T165 连续且无重复。
- 所有 FR、SC 均有任务映射。
- 没有未解释的 CRITICAL/HIGH 问题。
- T002/T003 的依赖和授权边界正确。

### 4.3 C3：graphify 验证

在全部文档修改完成后执行一次：

```bash
graphify update .
```

定向查询：

- M0-G1～M0-G8。
- FR-031 和快照原因。
- Q13 幂等定义。
- 模板反思和 `is_ai_added`。
- T002/T003 双 Agent 流程。

诊断要求：

- 无缺失端点。
- 无悬空边。
- 无自环。
- 无重复边。
- 无旧单分支或旧审查报告关系。
- 图谱异常收缩时停止覆盖并调查。

### 4.4 C4：写入修复证据

将实际命令、结果、模板哈希和图谱统计写入联合审查报告的“修复验证记录”。

阶段出口：

- M0-G1、G2、G4、G5 具备关闭证据。
- M0-G3 完成最终复核。
- M0-G6 具备关闭证据。
- M0 仍为 `in_progress`，因为 G7 和 G8 尚未完成。

## 5. 阶段 4：任务 D1——P2 历史隐私清理

状态：必须等待单独授权。

执行前确认：

- 清理对象和历史范围。
- 所有本地、远程分支及标签。
- 协作者停止推送窗口。
- Fork、旧克隆、缓存和 CI 产物处理。
- 强制更新远程的方式。
- 当前普通文档修改如何在清理后保留。

建议：

- 使用专用镜像克隆。
- 不在当前脏工作区直接试验。
- 不在报告中复述敏感身份信息。
- 改写后重新克隆并扫描。
- 核验所有分支、标签和远程引用。

阶段出口：M0-G7 关闭。

## 6. 阶段 5：任务 D2——P2 形成 G8 共享基线

状态：必须单独获得提交和推送授权；历史清理授权不自动包含本步骤。

历史清理完成后：

1. 在清理后的 `main` 恢复阶段 1～3 验证通过的文档修改。
2. 重新执行链接、模板哈希、Spec Kit 和 graphify 验证。
3. 更新联合审查报告的最终证据。
4. 将 M0 标记为 `complete`、M1 标记为 `ready`。
5. 形成单一目的 docs-only 基线提交。
6. 记录最终提交 ID。
7. 经授权推送并核对远程状态。

阶段出口：M0-G8 关闭，M0 完成。

## 7. 阶段 6：P3 启动 M1

状态：必须再次获得分支和 Issue 操作授权。

执行：

1. 从同一最终 `main` 提交创建 `codex` 和 `trae`。
2. 两个分支记录相同基线 ID。
3. 创建一个 M1 共享父 Issue。
4. 创建 Codex、Trae 两个独立实现子 Issue。
5. 两个 Agent 分别编写最小实施计划。
6. 两个实现分支不得互相合并或 cherry-pick 实现代码。

## 9. 停止条件

出现以下任一情况应暂停并报告：

- canonical 文档之间出现新的业务冲突。
- `.docx` 哈希或版式意外变化。
- Spec Kit 出现新的未解释 CRITICAL/HIGH 问题。
- graphify 异常大幅收缩或出现悬空关系。
- 发现新的密钥或个人信息。
- 执行步骤需要尚未获得的提交、推送、分支、Issue 或历史改写授权。
