# ADR-0011：增加密码与 TOTP 双因素备用登录

日期：2026-07-23

状态：已接受

## 背景

ADR-0010 选择 WebAuthn 通行密钥作为唯一常规登录，以消除密码、撞库和人工输入 OTP
形成的弱兜底。但用户在更换、重置、损坏或暂时无法使用原设备时，可能没有可用通行密钥。
现有离线恢复码加人工核验是紧急账号恢复，不适合作为正常设备切换流程；每次换设备都启动
人工恢复会造成高运营成本，也会诱使用户把恢复码放在不安全的位置。

本决策只回答“已有已激活用户如何在通行密钥暂不可用的新设备进入系统并新增通行密钥”。
它不改变受限公网入口、邀请激活、最后管理员、会话 Cookie 或生产部署延后等既有边界。

## 决策

### 1. WebAuthn 仍是首选登录，新增独立双因素备用登录

- WebAuthn 通行密钥仍是首选且具备 RP/Origin 绑定钓鱼抗性的登录方式。
- 备用登录必须在同一次登录意图中同时验证账号标识、密码和 TOTP；密码与 TOTP 任一项均
  不得单独建立会话。
- 不增加短信、邮件验证码、微信小程序 MFA、知识型问题、默认管理员或万能恢复材料。
- 密码与 TOTP 都不宣称具备钓鱼抗性。界面应明确推荐用户在新设备补建通行密钥。

### 2. 管理员强制配置，教师可选

- 管理员通过首个通行密钥完成激活后，必须在受限设置会话中建立密码和 TOTP，完成前不能
  访问业务页面。
- 教师可以选择启用或关闭；未启用不阻止业务使用，但本人安全设置持续显示可跳过提示。
- 管理员角色新增、移除或恢复后必须重新计算强制门禁；管理员不得关闭备用登录。
- 只有密码合格且首个 TOTP 成功验证后，两项材料才原子生效；不得保存半启用配置。

### 3. 密码与 TOTP 的固定安全参数

- 密码接受 8–128 个 Unicode 字符与空格，允许粘贴和密码管理器；不设字符组合规则、不要求
  周期轮换。建立或修改时拒绝常见、已泄露、可预期和与账号/产品相关的密码。
- 密码使用 Argon2id 和独立随机盐保存，只保留 PHC 摘要；成功验证后可按当前参数渐进重摘要。
- 每账号 TOTP 种子唯一且高熵，使用 6 位数字、30 秒周期；允许当前时间步及相邻一个时间步
  的合理偏差，同一或更早的已成功计数器不得再次接受。
- TOTP 种子只在绑定响应中展示一次，同时提供二维码和人工输入值。服务端以
  AES-256-GCM 加密，随机 96 位 nonce，AAD 绑定园所、用户、凭据和信封版本；主密钥位于
  数据库和代码库之外并通过 `key_id` 支持轮换。
- 密码、摘要、TOTP、种子、二维码和完整认证请求不得进入 URL、Cookie、Redis、日志、
  异常、审计或测试快照。

### 4. 备用登录建立普通业务会话

- 密码与 TOTP 同时正确时建立可撤销普通业务会话，API 仍按账号、角色、园所和班级授权。
- 会话必须持久化认证方式和备用因素版本；备用材料变化或恢复后，旧备用会话下一请求失效，
  仍有效的 WebAuthn 会话不因备用因素轮换而失效。
- 未知账号、未配置、密码错误、TOTP 错误、过期或重放使用相同公开响应和失败分类。
- 按可信客户端来源、账号标识摘要和端点全局实施相互独立的限流与渐进等待；备用限流不能
  无关阻断通行密钥登录，也不能由通行密钥成功清零。
- 备用登录成功属于安全事件，向用户提供不含秘密的内建通知并记录最小化审计。

### 5. 备用会话只可经专用再验证新增通行密钥

- 备用会话可以使用角色允许的普通业务，但不能被当作 WebAuthn 重新验证。
- 用户最近五分钟再次同时验证密码与一个未重放 TOTP 后，只可为当前用户新增通行密钥。
- 该证明不能重命名、删除或撤销通行密钥，不能修改密码/TOTP、轮换恢复码、管理账号/
  角色/邀请或批准恢复。
- 新通行密钥注册成功不自动提升既有会话；用户必须用新通行密钥重新验证或重新登录。

### 6. 因素维护继续要求 WebAuthn

- 仍有有效通行密钥时，建立、重设或关闭备用因素必须先完成最近五分钟 WebAuthn 重新验证。
- 教师可关闭，管理员不得关闭。密码/TOTP 替换或关闭必须使旧材料、全部备用会话和未消费
  专用证明立即失效。
- TOTP 换设备采用新种子重新绑定；新种子确认后不保留旧种子宽限期。
- 页面只显示是否启用、管理员强制状态和最近修改时间，不显示摘要、种子或离线验证材料。

### 7. 紧急恢复不降级

- 若仍有通行密钥，用户使用 WebAuthn 重新验证来重设备用因素。
- 若所有通行密钥均不可用，密码遗忘或 TOTP 丢失都不能引入新捷径；继续要求离线恢复码加
  人工核验，最后管理员继续要求预登记园所负责人和独立运维/安全责任人双人核验。
- 紧急恢复完成后撤销旧密码、TOTP、通行密钥、会话、邀请和恢复码。管理员绑定新通行密钥
  后必须重新配置备用登录；教师可以跳过。
- 管理员不得为其他用户直接设置或重置密码、TOTP 或通行密钥。

## 备选方案

### 密码单独登录

不采用。它会重新引入撞库和单因素接管路径，不能满足本次用户确认的双因素要求。

### 短信或邮件验证码

不采用。它引入外部交付依赖、SIM 换卡/邮箱接管和生产运营成本，也没有解决离线可用性。

### TOTP 单独登录或作为通行密钥的第二因素

不采用。TOTP 单独强度不足；强制把 TOTP 加在每次通行密钥登录上会降低首选路径可用性，
但不能提供比 WebAuthn 更强的钓鱼抗性。

### 备用会话只能补建通行密钥，不能使用业务

不采用。用户已经明确选择独立备用登录；两项因素同时验证后可以进入普通业务，但高风险
身份操作仍按认证保证级别隔离。

### 备用会话允许管理全部身份材料

不采用。密码与 TOTP 均可被钓鱼；若允许删除旧通行密钥或重设恢复材料，备用路径会成为
完整账号接管入口。

### 所有通行密钥丢失时由管理员重置备用因素

不采用。该路径弱于既有恢复双条件，并会绕过最后管理员职责分离。

## 后果

正面影响：

- 用户更换设备时可在无需人工恢复的情况下进入系统并补建通行密钥。
- 管理员具有预先验证的第二条登录路径，减少单设备失效造成的运营中断。
- 认证方法与保证级别显式建模，普通业务和高风险身份操作不再依赖“是否已登录”的模糊判断。

代价与风险：

- 密码与 TOTP 不具备钓鱼抗性，增加撞库、在线猜测、TOTP 重放和共享种子数据库泄露风险。
- 系统必须运营密码阻断、Argon2id、TOTP 加密/轮换、限流、通知和新的会话撤销状态。
- TOTP 与通行密钥若同时位于完全失陷的设备，两个因素不能抵御该设备控制。
- 管理员强制绑定增加首次激活步骤；这是降低最后管理员单设备失联风险的有意取舍。

- M3 Issue #7 的 T036–T045 与 `0004_settings.py` 保持不变。M3A 排在 M3 后、M4 前，

## 实施与验证约束

- 本 ADR 本身不授权业务实现。Issue 必须固定包含本 ADR 的已确认 docs 提交，只在 `dev`
  实施并执行 RED → GREEN。
- 必须覆盖密码策略/摘要、TOTP 时间窗口/重放、AEAD 标签失败、园所隔离、账号枚举、三层
  限流、管理员门禁、专用升级证明、会话撤销、紧急恢复和敏感信息扫描。
- NiceGUI 仍只能通过 FastAPI API 使用身份能力；不得连接数据库或持有 TOTP 主密钥。
- 常规测试不得访问短信、邮件、真实外部认证或其他网络服务。
- Graphify 语义抽取可使用大模型；提交前必须记录实际模型/代理来源并验证生成图不包含
  密码、TOTP 种子、二维码或其他秘密。

## 复审触发条件

出现企业身份提供商、托管设备、监管认证等级、TOTP 钓鱼/接管事件、管理员绑定完成率不可
接受、备用登录导致显著攻击流量，或密码/TOTP 的运营成本超过收益时，通过新 ADR 复审。
不得静默降低为单因素，也不得以短信/邮件或管理员代重置替换现有边界。
