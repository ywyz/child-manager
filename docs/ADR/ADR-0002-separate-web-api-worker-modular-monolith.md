# ADR-0002：采用独立 Web、API、Worker 运行单元的模块化单体

日期：2026-07-11

状态：已接受

## 背景

项目首期是一日活动计划，随后会接入游戏观察、一对一倾听等子系统。旧系统采用 NiceGUI 页面直接调用业务服务的一体化方式，初期简单，但容易让 UI、数据库、AI 和 Word 操作互相耦合。与此同时，首期规模不足以支撑每个业务模块拆成独立微服务。

## 决策

1. NiceGUI Web、FastAPI API 和后台 Worker 从首期开始就是三个独立部署进程。
2. Web 只负责页面、交互和反馈，只能通过 API 使用业务能力，不连接数据库或导入后端业务实现。
3. API 负责认证授权、输入校验、事务边界、园所隔离和同步业务编排。
4. Worker 负责 AI、Word 导出等耗时任务，不接受公网请求。
5. API 与 Worker 共享同一后端领域与应用代码包，共用 PostgreSQL；首期不是独立数据库微服务体系。
6. `packages/contracts` 只保存稳定的跨进程请求、响应、任务模型和枚举，不能包含 ORM、Repository 或供应商客户端。
7. 后续子系统优先作为模块加入同一后端，达到真实拆分条件后再独立成服务。

## 备选方案

### NiceGUI 一体化进程

不采用。它会弱化服务端权限边界，使页面事件容易直接访问数据库、AI 和文件系统，也不利于独立扩展 Worker。

### 前端直接访问数据库或共享 Repository

不采用。这会绕过 API 的权限、审计和事务控制。

### 从首期开始全面微服务化

不采用。独立数据库、服务发现、分布式事务和多服务发布的成本超过当前容量与团队需求。

### React/TypeScript 独立前端

当前不采用。NiceGUI 能保持 Python 技术栈并满足首期界面；服务边界已经通过独立 API 保留未来替换前端的空间。

## 后果

正面影响：

- UI、业务、数据和耗时任务职责明确。
- Worker 可以独立扩容或重启。
- 后续子系统可以复用身份、设置、提示词、任务和审计能力。
- 将来替换 Web 技术不会要求重写核心业务。

代价与风险：

- 即使都使用 Python，也要维护 API 和任务契约。
- 开发环境需要同时运行多个进程。
- Web 与 API 的认证 Cookie 转发必须明确实现和测试。

## 实施约束

- 通过静态检查或依赖测试禁止 `apps/web` 导入 ORM、Repository 和后端业务包。
- API 与 Worker 必须运行同一代码和契约版本。
- 不为每个类机械创建接口；只在真实边界上抽象。

## 复审触发条件

某个模块出现独立扩缩容、独立数据生命周期、法规隔离或独立团队发布需求时，可以提出拆分服务的新 ADR。
