# ADR-0001：只交付 Cloud 版本，首期单园部署并保留园所边界

日期：2026-07-11

状态：已接受

## 背景

旧系统曾规划本地版与未来 Cloud 版，导致存储、身份、文件路径和同步边界需要同时兼顾。新项目的目标是尽快交付可在线使用的幼儿园教育管理系统，首期实际只服务一所幼儿园，但后续可能扩展到多园部署。

如果完全按单园数据建模，未来增加多园能力时需要重构教师、班级、学期、教案和审计等核心关系；如果首期直接建设多园 SaaS 运营平台，又会引入当前不需要的租户管理、计费和跨园运营复杂度。

## 决策

1. 产品只交付 Cloud 版本，不开发本地版、桌面版、离线模式或本地到云端同步。
2. 首期一个生产部署只服务一所幼儿园，不提供多园 SaaS 管理界面。
3. 所有园所范围业务表保留 `kindergarten_id`，包括教师、班级、学期、班级区域、提示词配置、教案、任务、导出和审计等数据。
4. Repository 的查询、写入和唯一性校验必须显式携带 `kindergarten_id`；API 从服务端身份上下文取得该值，不把客户端参数作为权限依据。
5. 首期不实现跨园切换、平台管理员、计费或园所自助开通。

## 备选方案

### 同时维护本地版和 Cloud 版

不采用。两套运行环境会扩大认证、存储、升级和测试矩阵，并产生尚无业务价值的数据同步问题。

### 首期完全按单园建模

不采用。短期字段更少，但未来几乎所有核心表和唯一约束都要重构。

### 首期直接建设多园 SaaS

不采用。当前没有跨园运营需求，不应提前引入平台级角色、租户生命周期和计费系统。

## 后果

正面影响：

- 只有一条产品和部署路线。
- 数据模型保留未来多园隔离的必要边界。
- 单园环境仍可使用简单的 Docker Compose 部署。

代价与风险：

- 首期查询和约束必须始终携带看似重复的 `kindergarten_id`。
- 仅有字段并不等于已经具备多园 SaaS 能力；未来仍需补充租户生命周期、平台权限和运维隔离。
- 实现遗漏园所条件可能造成严重数据泄露，必须以 Repository 和 API 集成测试持续验证。

## 实施约束

- 园所范围表至少包含 `kindergarten_id`、`created_at`、`updated_at`。
- 唯一约束必须包含园所维度，例如教案使用 `(kindergarten_id, class_id, plan_date)`。
- SQLite 测试不得掩盖 PostgreSQL 中的园所约束与并发行为。

## 复审触发条件

只有在产品正式需要一个部署服务多所幼儿园时复审。届时应新增 ADR，而不是直接把本记录改写为“多园 SaaS 已完成”。
