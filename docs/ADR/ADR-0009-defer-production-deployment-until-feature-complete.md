# ADR-0009：功能完成前延后生产部署与访问网络决策

日期：2026-07-12

状态：部分被 [ADR-0010](ADR-0010-restricted-public-entry-passkey-authentication-and-recovery.md) 取代

取代范围：访问网络方向、NiceGUI Web/BFF 唯一公网应用入口、WebAuthn 常规认证及紧急恢复不再等待功能验收后选择。生产部署实施、反向代理产品、域名证书、拓扑、生产密钥托管、备份、发布和恢复演练继续延后。

## 背景

原架构将 Linux、Docker Compose 和 Caddy 公网 HTTPS 入口作为首期已确认的生产部署方案。实际公网系统上线后会持续遭受自动化扫描，生产入口是否改为 Tailscale 等私有网络、如何实施设备准入和应急恢复，尚需在完整功能和安全需求可验证后评估。

在功能开发阶段同时固化生产网络和部署细节，会提前引入与当前业务闭环无关的运维工作，也可能让未经安全复审的公网方案被误用。

## 决策

1. 身份、设置、手工教案、AI 生成、Word 导出等首期功能完成前，不实施生产部署。
2. 功能开发阶段不创建生产 Caddyfile、生产 Compose 拓扑、公网 DNS、证书、端口映射、Tailscale 配置或云主机发布流程。
3. 公网、Tailscale 等私有网络、统一 HTTPS 入口的实现方式与应急管理通道均标记为待复审，不再把 Caddy 公网入口视为已确认方案。
4. Web、API 和 Worker 的服务边界、同源 Cookie 安全结果、服务端授权、数据隔离与密钥不入库等安全要求继续有效。
5. 允许为本地开发和自动化测试提供最小依赖启动方式，例如测试用 PostgreSQL 和 Redis；这不等同于生产部署设计。
6. 所有首期功能完成并通过功能验收后，先进行威胁建模和访问网络评审，再以新 ADR 确认生产拓扑、入口、密钥、备份、恢复和上线流程。

## 影响

- ADR-0007 中已确认的 Caddy 公网入口、生产 Compose 拓扑和立即实施时序被本记录取代。
- ADR-0007 中“密钥不进入代码库、数据库、镜像或日志”等安全结果仍为强制要求，具体托管方式延后决定。
- 当前实施路线以可测试的完整业务功能为终点，不以可公网访问或可生产发布为阶段完成标准。
- 功能完成后仍必须执行生产安全、备份、恢复和访问控制设计；本决策只是延后，不是取消。

## 复审触发条件

首期所有功能完成并通过功能验收后触发复审。复审至少必须比较私有网络专用访问、受限公网入口和带外恢复通道，并输出威胁模型与可验证的防火墙规则。
