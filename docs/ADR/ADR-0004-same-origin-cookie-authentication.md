# ADR-0004：采用同源入口、HttpOnly Cookie 与 API 统一授权

日期：2026-07-11

状态：已接受

## 背景

NiceGUI Web 与 FastAPI API 独立部署，但用户应只感知一个网站。系统涉及教师、教案、AI 配置和未来幼儿照片，不能把访问令牌暴露给浏览器脚本，也不能让 Web 隐藏按钮成为权限控制。

## 决策

1. 浏览器只通过 Caddy 的统一 HTTPS 域名访问系统。
2. API 是身份验证、会话撤销、角色、园所和班级权限的唯一权威。
3. 用户使用用户名或手机号加密码登录，密码使用 Argon2id 哈希。
4. API 使用短期 access token 与可撤销、可轮换的 refresh token。
5. 令牌只放在 `Secure`、`HttpOnly` 且设置合适 `SameSite` 属性的 Cookie 中，不进入 JavaScript、`localStorage`、日志或 NiceGUI 持久化存储。
6. NiceGUI Web 服务端调用 API 时转发当前请求的认证 Cookie；API 对每次请求重新授权。
7. 改变状态的请求同时采用 SameSite、来源校验和 CSRF 防护。

## 备选方案

### 浏览器使用 localStorage 保存 Bearer Token

不采用。脚本注入风险会直接暴露长期会话凭证，也增加前端令牌刷新逻辑。

### Web 自行维护独立权限系统

不采用。Web 与 API 的角色结论可能漂移，且后台接口仍可能被越权调用。

### 纯服务端内存 Session

不采用。它会给 Web/API 多进程和未来水平扩展带来粘性会话与状态共享问题。

### 首期接入第三方身份平台

不采用。当前只有管理员与教师两类内部账号，没有单点登录或社会化登录需求。

## 后果

正面影响：

- 浏览器脚本无法读取令牌。
- API 始终掌握权限和撤销状态。
- 同源部署减少 CORS 和跨站 Cookie 配置复杂度。

代价与风险：

- Web 必须正确转发 Cookie 和 `Set-Cookie` 结果。
- Cookie 认证必须明确处理 CSRF、代理头和刷新令牌重放。
- 密码重置、账号停用和管理员操作需要撤销现有刷新令牌。

## 实施约束

- 不开放公众注册；首位管理员通过一次性安全初始化创建。
- 不允许停用最后一个有效管理员。
- 生产环境强制 HTTPS，并对 Cookie 属性、CSRF 和退出撤销编写集成测试。
- 管理员查看全园数据不自动获得教师正文编辑与 AI 生成权限。

## 复审触发条件

需要企业单点登录、移动原生客户端、跨域开放 API 或外部系统授权时，通过新 ADR 评估 OIDC/OAuth 2.1 等方案。
