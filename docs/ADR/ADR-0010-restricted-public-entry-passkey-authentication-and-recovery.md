# ADR-0010：采用受限公网 B/S 入口、WebAuthn 通行密钥与双条件恢复

日期：2026-07-22

状态：已接受

## 背景

Child Manager 已确定为 Cloud-only B/S 系统，但 ADR-0009 曾把公网或私有网络、统一入口和应急恢复整体延后到功能完成后的生产复审。随着身份实现和生产安全设计推进，以下风险已经足以先冻结架构边界：公网扫描与自动化攻击、钓鱼和账号接管、邀请链接泄露、通行密钥丢失、最后管理员失去凭据、个人设备失窃或感染恶意软件，以及合法用户导出后的外传。

ADR-0004 已确认同源 Cookie、API 统一授权和 CSRF 等结果，但其第 3 项固定为用户名或手机号加密码登录。密码与密码重置会形成可钓鱼、可撞库且需要长期运营的第二套认证路径，不符合本次安全取舍。

本记录依据 [`docs/security/threat-model.md`](../security/threat-model.md) 冻结访问、认证、邀请和恢复边界。它不授权立即实施生产部署，也不选择反向代理、云厂商、证书、容器拓扑、生产密钥托管或备份方案。

## 决策

### 1. 继续采用 B/S，并采用受限公网入口

系统继续以浏览器/服务器（B/S）形式交付，权威身份、权限、数据和审计保存在服务端。

正式访问采用受限公网入口：

- 只提供一个受管 HTTPS Origin。
- NiceGUI Web/BFF 是唯一公网应用入口，浏览器业务请求必须先到 Web/BFF。
- FastAPI API、Worker、Redis、PostgreSQL、迁移、监控和运维端口不得拥有独立公网地址、端口或由公网入口直通的旁路。
- Web/BFF 通过私有网络调用 API；Web/BFF 不成为授权权威，API 仍对每个请求重新验证账号、角色、园所和班级权限。
- 公网入口必须具备默认拒绝、TLS、安全响应头、限流、请求上限、可信代理链和安全事件监控；具体产品及拓扑在生产部署 ADR 中决定。

“受限公网”不等于公开注册，也不表示任何知道域名的人可以绕过认证访问业务数据。

### 2. WebAuthn 通行密钥是唯一常规登录方式

- 用户只通过 WebAuthn 通行密钥完成常规登录，不设置或验证账号密码。
- 用户名、手机号和显示名可以继续作为账号标识与管理信息，但不是认证秘密。
- 注册与认证固定 RP ID，并精确校验允许的 HTTPS Origin；两种 ceremony 均要求用户验证（User Verification）。
- challenge 必须高熵、短时、单次使用并绑定预期 ceremony；服务端验证 type、challenge、Origin、RP ID hash、用户在场/验证标志、签名和凭据归属。
- 每个账号允许绑定多个通行密钥。新增或删除凭据及高风险管理员操作需要使用仍有效的通行密钥重新验证。
- 微信小程序 MFA 不进入首期认证关键路径。微信能力未来可以用于非权威通知或经新 ADR 评估，但不得成为 WebAuthn 的弱兜底。

### 3. 保留 ADR-0004 的会话与授权结果

WebAuthn 完成的是用户认证，不取代服务端会话：

- API 继续签发短期 Access Token 与可撤销、可轮换的 Refresh Token family。
- 令牌继续只放在 `Secure`、`HttpOnly`、合适 `SameSite` 的 Cookie 中，不进入 JavaScript、`localStorage`、日志或 NiceGUI 持久化存储。
- Web/BFF 负责原样转发必要 Cookie 与多值 `Set-Cookie`；API 负责会话轮换、撤销和每次请求授权。
- 状态变更继续强制 CSRF、Origin 校验和可信代理头约束。

因此，ADR-0004 仅密码登录、密码修改/重置及相关初始化方式被本记录取代；同源 Cookie、API 授权、CSRF、会话撤销和开发环境仅回环地址可用 `Secure=false` 等条款继续有效。

### 4. 首位管理员初始化及管理员单次邀请

- 空系统只允许通过本机或部署控制台的一次性初始化创建园所和待激活首位管理员，并发放首个单次邀请；不得提供公网初始化端点、默认管理员或初始密码。首位管理员激活后，该初始化入口永久关闭。
- 不开放公众注册。管理员先创建目标账号，再发放绑定该账号、园所、角色和签发人的单次邀请。
- 邀请令牌至少包含 128 位随机熵，只保存不可逆摘要，最长有效 24 小时，可撤销且重新签发立即作废旧邀请。
- 邀请秘密不得进入日志、审计正文、分析系统或 Referer。
- 邀请只允许目标用户注册首个通行密钥，不直接建立业务会话。
- 首位管理员由预先登记的园所负责人和部署责任人带外核验；其余账号由签发管理员或另一名有效管理员带外核验并确认激活。仅持有泄露邀请的人不能获得业务访问。
- 签发、使用、过期、撤销、激活和拒绝均记录最小化审计事件并通知相关管理员。

### 5. 离线恢复码加人工核验是唯一紧急恢复方式

- 首次激活后向用户签发至少 128 位随机熵的单次恢复码，只展示一次，建议离线保存；服务端只保存不可逆摘要。
- 恢复码校验必须限流。恢复码、人工核验或管理员批准任一项单独成立都不能建立登录会话。
- 普通用户恢复由有效管理员完成人工身份核验；恢复只授权撤销旧凭据并绑定新通行密钥。
- 恢复成功后撤销该账号全部旧通行密钥、会话、未使用邀请和旧恢复码，并签发新恢复码、通知用户、记录审计。
- 仍持有有效通行密钥的用户通过重新验证新增凭据，不使用紧急恢复路径。

### 6. 最后管理员采用双人带外核验

- 系统不得停用、降权或删除最后一名有效管理员，并应在正式使用前建立至少两名由不同人员持有的管理员账号。
- 当已无其他有效管理员可核验时，最后管理员恢复仍要求有效离线恢复码，并由预先登记的园所负责人和独立运维/安全责任人双人核验；两种责任不得由同一自然人兼任本次核验。
- 不设置默认管理员、万能恢复密码、维护后门或仅凭数据库改值的恢复方案。
- 双人核验只允许重新绑定通行密钥，不能直接导出数据、提升角色或跳过后续登录；完成后执行安全冷静期、全量通知和不可变审计。

具体核验材料、责任人登记、冷静期时长和生产操作步骤必须在上线前运行手册中确定并演练，但不得降低上述双条件与职责分离要求。

### 7. 明确个人设备和导出的残余风险

- 首期允许个人设备访问，但不把个人设备视为可信网络或授权依据。必须提供会话/通行密钥查看与远程撤销、会话超时、敏感操作重新验证、CSP 和最小浏览器持久化。
- 设备被完全控制后，恶意软件仍可能操纵已登录会话或读取页面内容；确认感染时必须撤销相关凭据与全部会话。
- 每次导出和下载都由 API 重新授权并审计，只提供短时、已认证下载，不创建公开永久链接。
- 合法用户下载后截图、复制或外传无法由 Web 应用彻底阻止。该风险由最小权限、审计、园所制度、人员教育和事件追责共同承担，首期不以 DRM 或破坏 Word 模板的水印机制伪装成完全防泄露。

## 备选方案

### 私有网络专用访问

本次不采用。它能减少公网扫描面，但个人设备接入、设备生命周期、ACL、紧急恢复和移动使用会引入额外运营负担。受限公网入口配合钓鱼抗性认证更符合当前用户范围；如果实际攻击、合规或设备管理要求变化，可通过新 ADR 重新评估。

### 密码加传统 MFA

不采用。密码继续承担撞库、重置、弱口令和钓鱼风险；可人工输入的 OTP 也不能提供与 RP 绑定的钓鱼抗性，并会形成长期弱兜底。

### 邀请链接直接激活账号

不采用。链接一旦从聊天、日志、浏览器同步或设备泄露，攻击者可抢先绑定凭据。单次、短时令牌仍需管理员带外人工确认。

### 恢复码单独恢复

不采用。离线材料可能被拍照、复制或与设备一起失窃；恢复码必须与人工核验共同成立。

### 微信小程序 MFA 作为首期第二因素或恢复渠道

不采用。它会把首期认证可用性和恢复安全依赖微信账号、平台接口和额外客户端，并可能形成弱于 WebAuthn 的关键路径。

### 默认管理员或运维后门

不采用。它会成为不可撤销、难审计且高价值的全局攻击入口。

## 后果

正面影响：

- 常规登录获得基于 RP/Origin 绑定的钓鱼抗性，不再运营密码和密码重置。
- 公网攻击面收敛到 Web/BFF，数据与后台运行单元保持私有。
- 邀请泄露、凭据丢失和最后管理员恢复都有明确的双条件与审计边界。
- 同源 Cookie、服务端授权、CSRF 和 Refresh Token 撤销等已实现安全结果可以保留。

代价与风险：

- 需要设计 WebAuthn ceremony、凭据生命周期、邀请状态、恢复码和双人核验流程，并迁移现有密码契约与数据模型。
- 部分旧设备或浏览器可能无法满足 WebAuthn 与用户验证要求，需要在实施前定义支持矩阵，不能通过启用密码兼容来绕过。
- 管理员人工确认邀请和紧急恢复会增加操作成本，但这是防止泄露材料直接接管账号的有意取舍。
- 受限公网仍会遭受扫描、拒绝服务和未知漏洞；生产入口必须经过独立安全验证。
- 合法用户下载后的外传与完全失陷终端仍是残余风险。

## 对现有决策和文档的影响

- [ADR-0004](ADR-0004-same-origin-cookie-authentication.md) 被部分取代：第 3 项密码登录及密码修改/重置、密码初始化相关后果和约束失效；其余同源 Cookie、API 授权、CSRF 和会话条款继续有效。
- [ADR-0009](ADR-0009-defer-production-deployment-until-feature-complete.md) 被部分取代：访问网络方向、唯一公网应用入口、常规认证和紧急恢复不再等待 M9 选择；生产部署实现、密钥托管、备份、发布和恢复拓扑仍继续延后。
- [ADR-0007](ADR-0007-caddy-compose-and-file-secrets.md) 不恢复生效。本记录确认安全边界，不确认 Caddy、Compose 或任何具体生产产品。
- [ADR-0001](ADR-0001-cloud-only.md) 的 Cloud-only、集中权威数据和单园边界继续有效；本记录完成其预留的访问网络后续决策。
- 现有规格、OpenAPI、数据模型和任务中的密码型认证内容不得作为新的认证实现基线。开始 `dev` 迁移前，必须另行完成这些详细文档和契约的 WebAuthn/邀请/恢复设计，对现有密码数据制定迁移与回滚方案，并由 Issue 固定新的 `docs` 提交 ID。

## 实施与验证约束

- 本次 docs-only 决策不授权生产部署、业务代码、迁移、依赖或 OpenAPI 修改。
- 后续实现必须先以测试固定 challenge、Origin/RP ID、UV、签名、重放、账号枚举、邀请泄露、恢复双条件、最后管理员和会话撤销行为。
- 不得为兼容旧实现保留隐藏密码登录、默认密码、万能恢复码或未经审计的数据库修复路径。
- 生产上线必须满足 [`threat-model.md`](../security/threat-model.md) 第 7 节门禁，并对唯一公网入口与所有私有运行单元执行外部探测。
- 常规测试不得调用真实短信、邮件、微信或付费外部服务，也不得记录真实身份或恢复材料。

## 复审触发条件

出现企业 SSO、原生移动客户端、托管设备强制准入、多园 SaaS、监管认证等级、通行密钥覆盖率不足、持续公网攻击超出承受范围，或最后管理员流程无法由实际组织执行时，通过新 ADR 复审，不得静默启用密码或 OTP 兜底。
