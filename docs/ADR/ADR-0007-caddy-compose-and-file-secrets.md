# ADR-0007：采用 Caddy、Docker Compose 与文件挂载 Secrets

日期：2026-07-11

状态：已接受

## 背景

首期在 Linux 服务器上单园部署，需要 HTTPS、多个 Python 进程、PostgreSQL、Redis、持久化导出和密钥管理。系统规模不需要 Kubernetes，但普通环境变量和仓库内 `.env` 不适合保存 AI Key 解密主密钥、令牌签名密钥等高敏感配置。

## 决策

1. 首期生产部署使用 Linux + Docker Compose。
2. Caddy 是唯一公网入口，负责自动 HTTPS、统一域名、路由和基础安全响应头。
3. Compose 至少运行 `caddy`、`web`、`api`、`worker`、`postgres`、`redis`，并提供一次性 `migrate` 和定时 `backup` 任务或等效宿主机任务。
4. 普通非敏感配置使用环境变量。
5. 数据库口令、令牌签名密钥和 AI Key 解密主密钥通过 Docker Secrets 兼容的只读文件挂载；API 与 Worker 读取同一主密钥文件。
6. 主密钥不进入代码仓库、数据库、镜像、普通环境变量或日志。
7. PostgreSQL、Redis、Worker 和管理端口只在私有容器网络中可访问。
8. 生产发布由管理员手动触发；迁移成功后才启动新版本 API 与 Worker。

## 备选方案

### Nginx

不采用。Nginx 能满足路由需求，但单园 Compose 下 Caddy 的自动证书管理和配置更简单。

### Kubernetes

不采用。当前容量和运维团队不需要编排平台的复杂度。

### 将所有 Secrets 放入 `.env`

不采用。普通环境变量容易被进程检查、诊断输出或错误的 Compose 文件处理暴露。

### 首期接入云厂商密钥管理服务

不采用为强制依赖。文件读取适配器保留未来接入空间，同时避免绑定具体云平台。

## 后果

正面影响：

- 单台 Linux 服务器即可形成可重复部署拓扑。
- HTTPS 证书运维成本较低。
- 主加密密钥与业务数据库、镜像和普通配置分离。

代价与风险：

- Docker Secrets 兼容文件仍依赖宿主机权限、备份和轮换纪律。
- 单机 Compose 不提供跨主机高可用，必须依赖备份和恢复目标。
- Caddy 配置、Cookie、安全头和代理头仍需上线前测试。

## 实施约束

- 容器使用非 root 用户并尽可能采用只读根文件系统。
- 仅指定临时目录和持久化卷可写。
- PostgreSQL、导出文件每日异地加密备份，至少保留 30 天；目标 RPO 24 小时、RTO 4 小时。
- 密钥读取封装为适配器，未来迁移云密钥服务不改变业务代码。

## 复审触发条件

出现多主机高可用、自动扩缩容、托管容器平台或组织统一 Nginx/云密钥规范时，通过新 ADR 替代相应部署决策。
