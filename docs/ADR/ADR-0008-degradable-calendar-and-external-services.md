# ADR-0008：日期与外部服务采用本地优先和软降级

日期：2026-07-11

状态：已接受

## 背景

教案日期需要计算学期周次、星期、季节和中国工作日。在线节假日接口和 AI 都可能发生网络、限流或供应商故障，但这些辅助服务不应阻断教师手工保存和导出。旧系统曾出现在线接口优先以及失败后推断普通工作日的方案，可能增加延迟或给出过度确定的结论。

系统没有天气服务，不能为生成效果而虚构实时天气。

## 决策

1. 日期、学期和季节规则由后端领域服务统一计算，Web 与 AI 不自行推断。
2. 学期开始日期所在自然周固定为第一周，此后每个周一递增。
3. 工作日以本地节假日库为主要来源，在线节假日 API 只作补充或校验，并缓存结果。
4. 本地库和在线 API 都无法判断时返回“无法确认”，不伪装为普通工作日。
5. 非工作日、学期范围外和无法确认均只显示软提示，不阻止填写、保存、AI 生成或导出。
6. 系统不接入天气服务，也不提供天气标签；AI 只接收按月份推断的季节和教师填写的信息。
7. AI 或在线节假日服务不可用时，手工编辑、保存、归档和已有导出下载继续可用。

## 备选方案

### 在线节假日 API 优先

不采用。日常判断不应依赖外网可用性，本地库延迟更低且更稳定。

### 外部服务失败时默认普通工作日

不采用。未知不等于工作日，错误的确定结论会误导教师。

### 非工作日禁止创建教案

不采用。幼儿园可能存在调休、临时活动或补充记录，系统只能提醒。

### 接入天气服务或让教师选择天气标签

不采用。一日活动计划没有已确认的天气数据需求，额外服务会增加隐私、故障和运维边界。

## 后果

正面影响：

- 主流程不依赖外部服务可用性。
- 日期结论具有明确的“工作日、非工作日、未知”语义。
- AI 不会收到虚构天气上下文。

代价与风险：

- 本地节假日库需要及时更新，并验证所覆盖年份。
- 在线来源和本地库冲突时需要诊断与缓存策略。
- 软提示意味着教师仍可能在错误日期保存，最终决定由教师承担。

## 实施约束

- 自动化测试不得真实调用在线节假日 API。
- 在线调用必须设置超时、缓存和脱敏诊断日志。
- 季节固定为 3–5 月春、6–8 月夏、9–11 月秋、12–2 月冬。
- 日期和自然周按 `Asia/Shanghai` 计算，时间点以 UTC 存储。

## 复审触发条件

只有园所正式提出校历覆盖、天气数据或更高准确度要求时，才新增 ADR 评估园所日历和天气服务；不得在现有记录中静默扩大范围。
