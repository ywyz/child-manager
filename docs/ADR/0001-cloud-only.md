# ADR-0001：只交付 Cloud 版本，首期单园运行并保留园所隔离边界

日期：2026-07-11

状态：已接受

## 背景

旧仓库 `ywyz/kindergartenManager` 曾同时兼顾本地零配置运行、Windows EXE、SQLite/MySQL 切换、云服务器部署和主子系统同步。该路线让身份、数据库迁移、文件路径、密钥、升级方式和测试矩阵同时承担本地与云端语义；旧仓库后期还曾以固定默认管理员和 `tenant_id=1`、`user_id=1` 维持单用户模式。这些做法降低了单机启动门槛，但把正式权限、多用户隔离和生产数据库差异推迟到了功能之后。

Child Manager 新项目需要一条明确的产品路线。首期实际服务一所幼儿园，教师和管理员通过浏览器共同使用系统；数据、任务和导出需要集中保存、统一授权和审计。与此同时，未来可能出现一个系统服务多所幼儿园的需求，如果首期完全按无园所维度的单租户模型建设，教师、班级、学期、提示词、教案、任务、导出和审计等核心关系都需要重构。

“Cloud-only”容易被误解为“必须开放公网访问”或“开发时不能在本机运行”。产品形态、访问网络和生产拓扑是三个不同决策：本 ADR 决定产品与数据权威边界；生产入口、Tailscale、反向代理、密钥托管和备份恢复按照 ADR-0009 延后复审。

## 决策

### 1. 产品只交付 Cloud 版本

Child Manager 只交付由服务端集中保存权威数据、通过浏览器访问的 Cloud 版本，不开发：

- 本地正式版或桌面安装版。
- 离线编辑、离线缓存后同步或本地与云端双向同步。
- SQLite 作为正式产品数据库的运行模式。
- 由浏览器或教师设备直接持有完整业务数据库、AI Key 或服务端权限上下文。

本地开发、自动化测试和浏览器调试不属于产品交付形态，可以在开发机运行服务；开发便利不能引入第二套正式产品语义。

### 2. 首期一个运行实例只服务一所幼儿园

首期不建设多园 SaaS 运营平台。一个正式运行实例在业务上只对应一所幼儿园，不提供：

- 园所自助注册、开通、停用或套餐计费。
- 平台管理员、跨园运营人员或跨园数据查询。
- 用户在多个园所之间切换的界面与会话语义。
- 跨园报表、共享资源库或园所间数据迁移。

首位管理员初始化后，由园内管理员维护教师、班级、学期和其他必要设置。单园运行不等于单用户模式，也不允许跳过登录、角色和班级授权。

### 3. 从首期保留园所隔离边界

所有园所范围业务表必须包含 `kindergarten_id`、`created_at` 和 `updated_at`。涉及教师、班级、学期、区域、AI 模型档案、提示词、教案、后台任务、导出和审计的数据关系必须保持同园约束。

Repository 的读取、写入、更新、删除和唯一性校验必须显式携带 `kindergarten_id`，禁止先读取跨园数据再在业务层过滤。API 从认证后的服务端上下文取得园所 ID，不将客户端提交的 `kindergarten_id` 作为授权依据。

数据库唯一约束和外键必须包含园所维度。例如同班同日教案使用 `(kindergarten_id, class_id, plan_date)` 唯一约束，并通过组合外键阻止跨园关联。全局枚举和 Alembic 迁移元数据可以例外，但应明确说明非园所数据属性。

这些边界只表示数据模型能够隔离园所，不表示多园 SaaS 已经完成。未来多园能力仍需另行设计租户生命周期、平台权限、资源配额、运维隔离和数据迁移。

### 4. Cloud 不等于公网开放

Cloud 版本要求服务端集中运行和数据集中管理，但不预先决定用户通过公网、私有网络还是其他受控入口访问。

根据 ADR-0009，在全部首期功能完成并通过功能验收前：

- 不创建或验收生产部署。
- 不创建生产 Caddyfile、生产 Compose、DNS、证书或端口映射。
- 不创建 Tailscale 或其他私有网络配置。
- 不把公网可访问作为任何功能里程碑的完成标准。

功能验收完成后，通过新 ADR 比较私有网络专用访问、受限公网入口和应急通道。采用 Tailscale 等私有网络不会违反 Cloud-only，只要权威服务和数据仍由集中服务端管理。

### 5. 生产数据语义统一采用 PostgreSQL

生产数据库为 PostgreSQL，结构只通过 Alembic 迁移。开发和自动化测试可以使用 SQLite，但业务代码、约束设计和验收不得依赖 SQLite 特性，也不得用 SQLite 测试替代 PostgreSQL 的组合外键、部分索引、GiST、并发和事务证据。

项目不提供让用户在设置页面切换 SQLite、MySQL 或 PostgreSQL 的产品能力，也不继承旧仓库的 MySQL 迁移链。

## 决策范围

本 ADR 确认：

- 产品只有 Cloud 形态。
- 首期为单园运行、多用户使用。
- PostgreSQL 是生产权威数据库。
- 园所隔离从首期进入表结构、Repository 和 API 授权边界。
- 本地开发环境不构成另一产品版本。

本 ADR 不确认：

- 生产云厂商、主机位置或地域。
- 公网或私有网络入口。
- Tailscale、反向代理、容器编排和域名证书。
- 生产密钥托管、监控、备份、RPO/RTO 和发布流程。
- 多园 SaaS 的运营、计费和平台权限。

前四项生产决策按照 ADR-0009 在功能验收后复审；多园 SaaS 仅在出现真实产品需求时复审。

## 备选方案

### 同时维护本地版和 Cloud 版

不采用。它会扩大认证、数据库、文件存储、升级、密钥和测试矩阵，并引入当前没有业务价值的同步与冲突解决问题。旧仓库为 SQLite、MySQL、Windows EXE 和云端部署同时修补迁移差异，是本项目不重复的风险案例。

### 本地优先，未来再迁移 Cloud

不采用。本地优先会鼓励固定身份、设备内权威数据和本地文件路径进入核心设计，使后续恢复登录、多用户协作、集中审计和 PostgreSQL 约束成为迁移项目。

### 首期完全按无园所维度的单园模型建设

不采用。短期可以减少字段，但未来会重构几乎所有核心表、唯一约束、Repository 接口和审计关系，也难以系统验证跨园数据泄露。

### 首期直接建设多园 SaaS

不采用。当前没有跨园运营需求，不应提前引入平台角色、租户生命周期、计费、配额和跨园运维复杂度。

### Cloud 版本必须直接开放公网

不采用。公网只是访问网络方案之一，且既有公网系统受到自动化扫描的经验表明，访问网络必须在功能完成后结合威胁模型重新选择。产品集中运行不要求默认暴露公网入口。

## 后果

### 正面影响

- 产品、身份、数据和升级只有一套正式语义。
- 多用户协作、服务端授权和审计从首期进入基础设计。
- PostgreSQL 约束能够作为跨园隔离和并发正确性的正式证据。
- 未来扩展多园时保留必要数据边界，同时避免提前建设 SaaS 运营平台。
- 私有网络或受限公网方案可以在功能完成后独立决策，不与产品形态耦合。

### 代价与风险

- 园所范围查询和约束必须携带看似重复的 `kindergarten_id`。
- 实现遗漏园所条件可能造成严重数据泄露，需要 Repository、API 和 PostgreSQL 集成测试持续验证。
- 系统不能在服务端不可达时提供离线编辑；可用性依赖未来生产拓扑和恢复设计。
- 单园实例仍需要完整身份、安全、备份和运维能力，不能因为只有一所园而降低要求。
- 保留园所字段不自动获得多园 SaaS 能力，未来仍有产品与运维设计成本。

## 实施与验证约束

- 园所范围模型、迁移、Repository 和 API 必须一起实现并测试。
- Repository 测试必须证明查询、写入、更新、删除和唯一性校验受 `kindergarten_id` 限制。
- API 测试必须证明客户端不能通过伪造园所 ID 越权。
- PostgreSQL 测试必须覆盖组合外键和关键唯一约束，SQLite 只能作为快速反馈环境。
- Web 不得直接连接数据库或自行决定园所上下文。
- 日志和审计只记录必要园所标识与元数据，不记录密钥或完整敏感正文。
- 当前不得增加本地正式版入口、数据库切换设置、同步队列或生产部署配置。

## 复审触发条件

出现以下任一情况时新增 ADR 复审，不直接改写本记录的历史决定：

- 产品正式要求离线工作、桌面安装或本地数据为权威来源。
- 一个运行实例需要同时服务多所幼儿园。
- 用户需要跨园身份、切换或数据访问。
- 法规、合同或数据驻留要求改变集中服务方式。
- 首期功能验收完成，需要决定生产访问网络和运行拓扑；该项按 ADR-0009 单独形成生产 ADR，不改变 Cloud-only 产品决定。
