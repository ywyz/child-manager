# ADR-0005：AI 供应商中立，并建立管理员专用提示词系统

日期：2026-07-11

状态：已接受

## 背景

一日活动计划需要文本生成，后续游戏观察和一对一倾听等子系统还可能使用视觉模型。模型供应商、模型名称、能力和价格会变化；提示词也需要由管理员持续调整。如果业务代码直接依赖供应商 SDK或把提示词散落在页面与服务中，后续模块会重复实现配置、版本、测试和审计。

## 决策

1. AI 通过统一 OpenAI 兼容接口接入，不在业务代码中硬编码供应商。
2. 管理员维护多个模型档案，包括 `API_BASE_URL`、加密 `API_KEY`、`MODEL_NAME`、能力标签、并发和限流参数。
3. 模型能力至少预留 `text`、`vision` 和 `structured_output`；任务只能选择满足能力要求的启用档案。
4. 提示词是系统级独立模块，后续所有子系统按稳定任务标识使用；只有管理员可以修改、发布和回滚。
5. 系统提供只读默认提示词，自定义提示词采用草稿、已发布版本、历史和回滚生命周期；业务只能读取已发布版本。
6. 输出 Schema 由代码固定，不允许管理员通过提示词管理页面改变业务数据结构。
7. 每次调用记录模型档案、模型名、提示词版本、耗时、状态和脱敏错误摘要，不记录 API Key 或完整敏感正文。
8. AI 结果必须通过结构化 Schema 校验并作为预览，未经教师明确采用不得覆盖人工内容。

## 备选方案

### 每个业务模块直接使用供应商 SDK

不采用。会产生重复错误处理、配置语义和供应商耦合。

### 只配置一个全局模型

不采用。后续文本与视觉任务可能需要不同模型能力、费用和限流策略。

### 将提示词写在代码或环境变量中

不采用。管理员无法安全调优、测试、回滚和审计，修改还需要重新发布应用。

### 允许提示词定义输出 Schema

不采用。运行时修改结构会破坏保存、校验、历史和 Word 映射链路。

### AI 直接写入教案当前内容

不采用。模型错误或重试可能覆盖教师已经编辑的内容。

## 后果

正面影响：

- 可以接入 OpenAI、DeepSeek、通义等兼容服务而不修改业务模块。
- 后续子系统复用统一模型、提示词、审计和错误分类。
- 教师保持对最终教案内容的决定权。
- 视觉能力已有模型档案和能力校验边界，但首期不提前部署照片系统。

代价与风险：

- 兼容接口仍可能存在供应商细节差异，需要由适配器归一化。
- 提示词版本、默认种子和测试运行增加数据模型与管理界面。
- 结构化输出仍会波动，必须实现 Schema 校验和限定重试。

## 实施约束

- AI API Key 使用服务端认证加密后保存，页面不回显明文。
- 常规自动化测试禁止调用真实或付费模型。
- 外部请求只发送当前任务必需内容，默认排除教师账号、手机号和幼儿身份信息。
- 一键生成按栏目拆分；集体活动只有教师提供原始教案后才能生成。

## 复审触发条件

只有在兼容接口无法表达必要的视觉、多模态或供应商能力时，才通过新 ADR 增加专用适配器；业务模块仍不得直接依赖供应商 SDK。
