# ADR-0003：PostgreSQL 保存任务权威状态，Dramatiq 与 Redis 负责异步执行

日期：2026-07-11

状态：已接受

## 背景

AI 生成和 Word 导出可能耗时较长、调用外部服务并发生重试，不能阻塞 API 请求。任务还必须在 Redis、Worker 或网络短时故障后恢复，并保证重复消息不会覆盖教师内容、重复创建快照或覆盖导出文件。

项目首期容量较小，不需要重量级事件平台，但仍需明确消息投递与业务状态的权威来源。

## 决策

1. 生产数据库使用 PostgreSQL；SQLite 仅用于开发和自动化测试。
2. 后台任务使用 Dramatiq 2.x，Redis 作为消息代理。
3. PostgreSQL 中的 `background_job` 是任务状态、业务调用次数和结果引用的唯一权威；Redis 不保存最终业务事实。
4. API 先提交 `pending_dispatch` 任务记录，再向 Redis 投递最小化的任务 ID 消息。
5. 投递失败时保留待投递状态，由恢复扫描器重新投递；超时运行任务通过租约、心跳和幂等规则恢复。
6. Worker 按任务 ID 和业务幂等键执行，基础设施重新投递不能突破 AI 首次调用加两次重试的上限。
7. 一键生成拆为栏目级子任务，单个失败不回滚其他栏目。
8. 首期 Web 每 1–2 秒短轮询 PostgreSQL 权威状态；完成、失败或离开页面后停止。暂不引入 WebSocket 或 SSE。

## 备选方案

### API 同步执行 AI 和 Word

不采用。长连接容易超时，也无法可靠处理重试和 Worker 独立扩容。

### Celery + Redis

不采用。Celery 能满足需求，但首期不需要其更广泛的调度和消息生态；Dramatiq 的能力与复杂度更符合当前范围。

### Redis 作为任务最终状态存储

不采用。Redis 数据丢失或清理后将无法审计、恢复和判断结果是否已采用。

### PostgreSQL 直接充当轮询任务队列

不采用。项目已经需要 Redis 处理任务投递和运行协调，使用成熟 Worker 库可以减少自建消费者协议。

### 立即引入事务消息中间件或 Kafka

不采用。首期通过数据库权威状态、提交后投递和恢复扫描即可达到所需可靠性。

### 首期使用 SSE 或 WebSocket 推送

不采用。短轮询更易断线恢复和调试；未来推送可以复用相同状态模型。

## 后果

正面影响：

- Redis 丢失后可以从 PostgreSQL 重建未完成队列。
- API 快速返回任务 ID，外部 AI 故障不会占用长请求。
- 栏目级隔离、幂等和审计边界明确。

代价与风险：

- 数据库提交和 Redis 投递不是单一原子事务，必须运行恢复扫描。
- 任务状态机、租约和幂等键需要专门设计与并发测试。
- 短轮询增加少量 API 请求，需要聚合子任务并限制异常频率。

## 实施约束

- Redis 消息不得包含 API Key、令牌或完整教案正文。
- Worker 必须从 PostgreSQL 读取当前权威上下文并验证资源版本。
- AI 结果先保存为预览；教师采用时由 API 再次鉴权、检查乐观锁并创建快照。
- Word 导出成功必须同时存在唯一文件存储键和导出记录。

## 复审触发条件

当短轮询负载显著影响 API、需要亚秒级进度，或任务规模超出 Redis/Dramatiq 的可运维范围时，通过新 ADR 评估 SSE、其他 Broker 或事务消息方案。
