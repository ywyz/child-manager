# Child Manager 双 Agent 独立开发协议

> **历史文档 / 已退役**：本协议记录 2026-07-12 至 2026-07-21 使用的 Codex/Trae 双实现流程。自 2026-07-21 起，当前开发遵循 [`single-implementation-development.md`](single-implementation-development.md) 的 `main/docs/dev` 单实现流程。本文只用于审计历史，不再授权或约束新的开发工作。

文档版本：v1.0

状态：已退役，仅供历史审计

日期：2026-07-12

适用对象：Codex、Trae、维护者与审阅者

## 1. 文档目的

本文定义 Codex 与 Trae 基于同一共享设计基线、分别完成两套独立实现时的分支边界、Issue 层级、规格同步、交叉评审、状态报告和验收规则。

双 Agent 的目标不是把一个实现拆给两个 Agent 拼装，而是让两方在相同产品约束下各自交付完整、可验证的系统。独立实现可以采用不同内部组织，但不得改变共享业务不变量、服务边界、安全要求和 Roadmap 门禁。

本文不决定最终采用哪套实现，也不授权任何 Agent 提交、推送、创建分支、Pull Request 或合并代码。

## 2. 事实来源与优先级

开展工作前依次阅读：

1. [`AGENTS.md`](../../AGENTS.md)：开发过程、安全、修改边界和验证规则。
2. [`CONTEXT.md`](../../CONTEXT.md)：当前仓库、分支与共同下一步。
3. [`README.md`](../../README.md)：产品范围、技术基线与分支职责。
4. [`ROADMAP.md`](../ROADMAP.md)：共享里程碑、依赖和出口门禁。
5. 当前任务对应的 PRD、ADR、设计、模板、迁移和测试。
6. 本文：双 Agent 的协作与交付协议。

本文只规定协作方式，不能覆盖上述文件中的产品或技术决策。来源冲突时按 `AGENTS.md` 停止会固化冲突的实现并请求确认。

旧仓库 `ywyz/kindergartenManager` 仅提供过程经验。可复用的经验包括分阶段验证、跨设备交接清单、明确记录已完成/已验证/已知问题/下一步，以及在真实 UI 闭环设置人工验收点；旧仓库的单用户模式、部署顺序、技术栈和具体实现不构成本项目需求。

## 3. 分支模型与所有权

| 分支 | 所有者 | 允许内容 | 禁止内容 |
| --- | --- | --- | --- |
| `main` | 维护者共享管理 | 根目录说明、`docs/`、共享 `specs/`、模板、架构约束和变更记录 | 业务实现、迁移、实现专属测试和生产部署文件 |
| `codex` | Codex | Codex 的完整实现、迁移、测试和分支内实现说明 | Trae 实现代码、未经确认的共享规格改写、生产部署文件 |
| `trae` | Trae | Trae 的完整实现、迁移、测试和分支内实现说明 | Codex 实现代码、未经确认的共享规格改写、生产部署文件 |

执行边界：

- `codex` 与 `trae` 必须从同一个经确认的 `main` 基线提交创建，并记录该提交 ID。
- 两个实现分支不得互相合并、cherry-pick 实现提交、复制补丁或以另一方代码作为自己的实现起点。
- 不得自行把任一实现合并回 `main`。
- `main` 的共享文档与模板更新使用单一目的提交，再分别 cherry-pick 到两个实现分支。
- 两个 Agent 并行工作时必须使用不同 Git worktree 或独立克隆，不得在同一工作目录中反复切换分支或共享未提交文件。
- 同机并行启动必须遵循 [`local-development-environments.md`](local-development-environments.md) 的端口、Compose 项目、数据库和运行时目录隔离约定。
- 实现分支自己的进度、实验记录或内部设计不得反向写入共享规格，除非它揭示了共同需求缺口并按第 7 节完成确认。
- 未经明确授权，Agent 不得切换或创建分支、提交、推送、创建 Pull Request 或改写历史。

## 4. Issue 层级与状态隔离

每个 Roadmap 里程碑采用一个共享父 Issue 和两个实现子 Issue：

```text
M<n> 共享里程碑 / 设计父 Issue
├── M<n> Codex 实现子 Issue
└── M<n> Trae 实现子 Issue
```

### 4.1 共享父 Issue

父 Issue 只记录：

- 共同目标、入口条件和出口门禁。
- 已确认 PRD、ADR、设计与模板链接。
- 共同业务不变量和可观测验收结果。
- 用户确认的共享决策及对应文档提交。
- 两个实现子 Issue 的链接和各自状态。

父 Issue 不记录某一实现的类名、数据库访问细节、临时调试过程或专属提交列表。只有两个实现子 Issue 都满足共同门禁时，父 Issue 才能标记完成；一方完成不得代表另一方完成。

### 4.2 实现子 Issue

每个子 Issue 独立记录：

- 所属分支与共享基线提交。
- 对应父 Issue、规格版本和依赖项。
- 该实现的最小方案与明确非目标。
- 可执行验收标准、测试命令和人工冒烟路径。
- 实际提交、迁移、验证结果、已知风险与阻塞。

子 Issue 只更新自身实现状态。Agent 不得代替另一方关闭 Issue、勾选验收项或推测完成度。

### 4.3 纵向拆分

里程碑进入 `ready` 后才能拆实现 Issue。实现任务优先形成端到端可验证的纵向切片，例如“管理员创建教师账号并由 API 校验权限”，而不是按“全部模型”“全部 Repository”“全部页面”进行长期水平拆分。

设计问题与实现工作默认分开：共享设计先在父 Issue 和 `main` 文档中确认，再由两个实现子 Issue 独立落地。

## 5. 每个里程碑的协作流程

### 5.1 准备

1. 确认上一里程碑已经满足 Roadmap 出口门禁。
2. 确认两个分支都包含同一版共享文档和模板提交。
3. 创建或核对共享父 Issue，以及 Codex、Trae 两个实现子 Issue。
4. 为两个实现准备独立 worktree；需要同机并行运行时，分别加载 Codex、Trae 本地档位并检查目标端口未被其他进程占用。
5. 两个 Agent 分别写出最小实现计划，并为每一步定义验证方式。
6. 若计划暴露共享规格缺口，先执行第 7 节，不开始受影响实现。

### 5.2 独立实现

- 两个 Agent 只在各自分支工作，各自维护迁移链、依赖锁和测试。
- 允许内部架构不同，但 Web/API/Worker 边界、园所隔离、权限、任务幂等、AI 数据最小化和 Word 模板要求必须一致。
- 不以另一方已经完成作为省略测试、迁移或负向验证的理由。
- 不为追求表面一致而同步内部代码结构；只同步已确认的共享契约和用户可观察行为。
- 当前阶段不设计或创建生产部署内容。

### 5.3 分支内验证

每个 Agent 先执行与修改最相关的快速测试，再执行风险相称的完整验证。工程骨架建立后至少运行：

```bash
uv sync --locked
uv run ruff format --check .
uv run ruff check .
uv run pyright
uv run pytest
```

涉及 PostgreSQL、Alembic、Redis/Worker、Word 或浏览器流程时，还必须执行 Roadmap 对应专项检查。未执行的检查必须说明原因，不能写成通过。

### 5.4 只读交叉评审

一个实现子 Issue 完成自身验证后，可以请求另一 Agent 进行只读交叉评审。交叉评审允许：

- 阅读对方分支的差异、测试、迁移和验证证据。
- 对照共同 PRD、ADR、设计与父 Issue 检查行为偏差。
- 指出遗漏的权限、隔离、并发、幂等、安全、模板或降级场景。
- 在对方实现子 Issue 中提交可执行的评审意见。

交叉评审禁止：

- 修改、提交或推送对方分支。
- 复制对方源代码、测试、迁移或补丁到自己的实现。
- cherry-pick 或合并对方实现提交。
- 把“对方已测试”当成自己分支的验证证据。
- 要求两方内部类名、目录或实现模式机械一致。

评审意见分为：

- `共享规格问题`：按第 7 节处理，可能同时冻结两方相关工作。
- `共同门禁偏差`：由被评审方在自己的分支修复并重新验证。
- `实现选择差异`：不违反规格和门禁时无需统一。
- `建议优化`：不影响当前验收时不得借机扩大范围。

### 5.5 里程碑验收

每个实现分别提交自己的验收证据。父 Issue 以并列表格记录两方结果，不用测试数量或代码量排名：

| 共同门禁 | Codex 证据 | Trae 证据 |
| --- | --- | --- |
| 功能验收项 | 独立链接与结果 | 独立链接与结果 |
| 标准质量命令 | 实际命令与结果 | 实际命令与结果 |
| 专项负向测试 | 独立证据 | 独立证据 |
| 浏览器冒烟 | 独立步骤与结果 | 独立步骤与结果 |
| 已知风险 | 分支专属风险 | 分支专属风险 |

一方被阻塞时，另一方可以继续不依赖该阻塞的工作，但父里程碑仍不能完成。

## 6. 共享文档与模板同步

共享规格只在 `main` 维护。同步流程为：

1. 在 `main` 完成确认后的文档或模板修改。
2. 形成只包含根目录说明文件、`docs/`、共享 `specs/` 或 `templates/` 的单一目的提交。
3. 记录提交 ID、变更原因、影响里程碑和是否需要实现调整。
4. 经授权后分别 cherry-pick 到 `codex` 与 `trae`。
5. 两个实现子 Issue 分别记录已同步的提交 ID 和影响评估。
6. 同步后重新运行受影响的验证；不得只确认 cherry-pick 成功。

优先保留共享文档提交的历史关联。只有无法合理 cherry-pick 且经明确确认时，才复制目录快照，并在提交中记录来源提交与差异原因。

模板同步还必须验证原始文件哈希和 Git 状态，禁止为解决合并冲突重建或重新排版 `.docx`。

## 7. 冲突、歧义与共同冻结

任一 Agent 发现事实来源冲突或高影响歧义时，应立即：

1. 停止会固化该决策的实现、迁移或契约修改。
2. 在共享父 Issue 记录冲突文件、章节、两种表述、影响范围和推荐答案。
3. 通知另一实现子 Issue，将受影响能力标记为暂停；不依赖该决策的工作可以继续。
4. 一次只向用户询问一个能够解除当前依赖的问题。
5. 用户确认后，先在 `main` 修正文档并形成共享提交。
6. 两个分支同步该提交并分别评估、调整和验证自己的实现。

任何一方不得以“先做一个版本”“之后再统一”为由静默选择。临时实验只能用于回答设计问题，不得作为未确认需求的正式实现提交。

## 8. 进度、交接与恢复工作

旧仓库的跨设备交接表明，单纯写“已完成”不足以安全续作。每个实现子 Issue 或分支交接记录至少包含：

- 当前分支、HEAD 提交和共享基线提交。
- 当前 Roadmap 里程碑、父/子 Issue 和规格版本。
- 已完成且已验证的行为。
- 实际运行的命令、结果和最近一次验证时间。
- 未提交工作区状态，但不复制敏感 diff 或秘密。
- 已知问题、失败原因和复现步骤。
- 下一条最小可执行任务及其完成标准。
- 是否需要真实 UI 手工验收，以及准确命令、URL 和期望结果。

恢复工作时先核对仓库事实、当前分支和 Issue 状态，不根据旧交接记录直接假定仍然有效。交接记录不得包含 API Key、令牌、Cookie、数据库口令、真实身份信息或完整敏感教案。

## 9. 进度报告与比较规则

两个实现统一使用 Roadmap 状态：`pending`、`ready`、`in_progress`、`blocked`、`complete`，不使用主观百分比。

比较两套实现时只比较共同验收维度：

- 需求与业务不变量符合度。
- 权限、园所隔离和安全边界。
- PostgreSQL 迁移、并发和事务正确性。
- AI 与后台任务的重试、恢复和幂等。
- Word 模板保真与文件一致性。
- 可维护性、故障诊断能力和验证可重复性。
- 用户流程和中文错误反馈。

不得以提交数、代码行数、测试数量、开发速度或与另一方内部结构相似度单独判定优劣。最终采用、合并或停止某套实现由用户另行决定，本文不预设胜出分支。

## 10. 禁止事项

- 不让两个 Agent 在同一实现分支同时写入。
- 不让两个 Agent 共用同一工作目录、Compose 项目名、宿主机端口、数据库、虚拟环境或运行时文件目录。
- 不把两个独立实现拆成互相依赖的半成品。
- 不在 `main` 解决实现分支的代码冲突。
- 不跨分支复制实现来“追平进度”。
- 不代替另一 Agent 更新状态、关闭 Issue 或声称测试通过。
- 不通过放宽断言、删除测试或隐藏失败制造一致结果。
- 不把旧仓库实现或另一 Agent 的实现升级为事实来源。
- 不在功能验收完成前创建生产网络、反向代理、Tailscale、备份或发布配置。
- 不在交叉评审中泄露或复制秘密、真实身份数据和敏感正文。

## 11. 启动前检查清单

开始 M1 工程骨架编码前，按以下顺序完成准备：

1. 完成 M1 共享父 Issue、Codex 实现子 Issue和 Trae 实现子 Issue 的草稿及只读复核。
2. 经 Issue 操作授权后创建或核对三个 Issue，并回填父子链接。
3. 经分支操作授权后执行 T003，从同一确认的 `main` 提交创建 `codex` 与 `trae`。
4. 两个实现子 Issue 分别记录相同的共同基线提交 ID。
5. 经实现授权后，两个 Agent 才分别在自己的分支开始 T004。

用户明确要求时，第 2、3 步可以交换，但 Issue 与分支操作仍需分别授权，并且都必须在编码前完成。

开始编码前，确认：

- [x] M0 共享设计基线已完成交叉审阅。
- [x] `main` 共享基线已形成经授权的提交。
- [x] `codex` 与 `trae` 获准从同一提交创建。
- [x] 两个分支记录了相同基线提交 ID。
- [x] M1 共享父 Issue #1 与实现子 Issue #2、#3 已建立。
- [x] 两个子 Issue 均包含可执行验收标准和非目标。
- [x] Agent 明确只能写自己的实现分支。
- [x] 双实现 worktree 与本地端口/数据隔离规则已形成共享文档。
- [x] 规格冲突、文档同步和只读交叉评审流程已被接受。
- [x] 生产部署仍处于 M9 延后状态。
- [x] M1 实现已获得明确授权。

全部满足后，两个 Agent 才分别进入 M1 工程骨架实现。
