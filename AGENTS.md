# Child Manager Agent 开发规则

本文件适用于仓库根目录及其所有子目录。实现分支可以在 `apps/web/`、`apps/api/` 等目录增加更具体的 `AGENTS.md`；局部规则只能细化当前规则，不得降低安全、数据隔离、业务不变量和验证要求。

这些规则强调正确性、可验证性和长期可维护性。对简单任务保持简洁，不要为了形式增加无价值步骤。

## 1. 事实来源与冲突处理

按内容所属领域使用事实来源，不要把某一个文件当作所有问题的唯一真相：

- `AGENTS.md`：开发过程、修改边界和验证规则。
- `docs` 分支中的 `docs/`、`specs/`、OpenAPI 契约和 `templates/`：产品需求、架构决策、稳定契约和模板的单一事实来源。
- `README.md`：产品概览、稳定版本状态和文档导航。
- GitHub Issue：执行范围、当前状态、阻塞与验收证据；Issue 不得改写已确认的产品或架构规则。
- `templates/teacherplan/teacherplan.docx`：Word 导出的实际版式。
- `templates/teacherplan/一日活动计划系统说明.md`：字段含义、内容格式和生成要求。
- Alembic 迁移与自动化测试：当前已实现行为的证据，但不能自行推翻已确认需求。

如果不同来源相互矛盾：

1. 明确指出冲突位置及其影响。
2. 停止会固化该冲突的实现。
3. 请求用户确认，不得静默选择或自行改写需求。

不得根据旧仓库 `ywyz/kindergartenManager` 的实现推断本项目需求。旧仓库只能作为参考；本项目采用 Python 3.14+、PostgreSQL、独立 NiceGUI Web 与 FastAPI API 服务等已确认方案。

## 2. 开始工作前

### 2.1 先理解再编码

- 先阅读根目录 `AGENTS.md`，再阅读 `README.md`、`CONTEXT.md`、当前任务在 `docs` 分支中的 PRD、ADR、架构、规格、契约、模板和对应 GitHub Issue；实现任务还要阅读迁移、测试和局部 `AGENTS.md`。
- 实现 Issue 必须固定引用已确认的 `docs` 提交 ID；不得只引用会继续移动的分支名。
- 明确说明会影响实现的假设；存在多种合理解释时不得静默选择。
- 能从仓库中查明的信息先自行查明，不向用户重复提问。
- 发现更简单且满足需求的方案时应主动说明。
- 需求不清、风险较高或会扩大范围时暂停并询问。

### 2.2 保持最小实现

- 只实现当前任务要求的能力。
- 不为单一用途创建无必要的抽象、工厂或配置层。
- 只有出现第二个实现、外部供应商边界或明确测试替身需求时才抽象适配器。
- 不为理论上不可能发生的情况堆叠防御代码。
- 如果实现明显可以更短、更直观，应先简化再提交。

### 2.3 先定义完成标准

多步骤任务开始前给出简短计划，并为每一步定义可执行的验证方式。例如：

```text
1. 增加校验 → 验证：无效输入测试失败后转为通过
2. 实现行为 → 验证：相关单元测试和 API 测试通过
3. 检查影响 → 验证：静态检查、类型检查和完整测试通过
```

修复缺陷时应先增加能够复现问题的测试，再完成修复。

## 3. 精准修改

- 只修改完成任务所必需的文件和代码。
- 不顺手重构、格式化或清理无关内容。
- 遵循当前分支已有风格；若风格本身是任务对象，再进行有边界的调整。
- 用户工作区中的已有修改属于用户，不得覆盖、回退或据为己有。
- 本次修改产生的无用导入、变量、函数和临时文件必须清理。
- 发现原有无关问题时可以报告，不得未经授权一并修改。
- 每一处变更都应能追溯到当前任务或其必要验证。

## 4. 分支与 Git 规则

长期分支职责：

- `main`：稳定版本与发布基线，只接收已经完成测试和 Review 的 `dev` 结果；当前迁移完成前仍是历史 docs-only 基线，不得提前宣称为可发布应用。
- `docs`：PRD、Architecture、ADR、Context、Development Guide、共享 `specs/`、OpenAPI 契约和 `templates/` 的单一事实来源。该分支可以包含从 `main` 继承的稳定代码快照，但提交不得修改业务代码、迁移、实现测试或依赖锁。
- `dev`：Codex 负责的唯一实现与集成分支，承载 Python 代码、数据库迁移、NiceGUI UI、测试和依赖锁。

执行规则：

- 所有业务实现必须由 GitHub Issue 驱动。Issue 必须包含固定的 `docs` 提交 ID、范围、非目标、验收标准和验证方式；缺少任一项时不得开始实现。
- 流程固定为 `Design -> docs 确认 -> Issue -> dev 实现 -> 测试 -> Review -> main`。实现发现需求缺口时，停止受影响工作，先修订并确认 `docs`，再更新 Issue 的文档基线。
- `docs` 的提交只允许修改根目录治理说明、`.github/ISSUE_TEMPLATE/`、`.specify/memory/constitution.md`、`.specify/templates/`、`docs/`、共享 `specs/`、OpenAPI 契约、`templates/` 或由工具生成的 `graphify-out/`；同步到 `dev` 后必须重新运行受影响验证。
- `dev` 不得反向修改已确认需求来迁就现有实现。`main` 禁止临时开发和未验收提交。
- 未经明确授权，不得提交、推送、创建 Pull Request、切换分支或改写 Git 历史。
- 禁止使用 `git reset --hard`、强制推送或其他破坏性命令。
- 提交应保持单一目的。Commit 使用 Conventional Commits 类型，主题中文优先，例如 `feat: 增加教案归档`。

## 5. 语言与命名

- 文档、Issue、架构说明和用户界面文本使用简体中文。
- 技术术语中文优先，首次出现时附英文名称；通用名称无需机械翻译。
- Python 模块、变量、函数、类、数据库字段、API 字段和错误码使用英文。
- Python 模块、函数、变量和测试文件使用 `snake_case`，类使用 `PascalCase`。
- 用户可见错误信息使用中文；稳定 API 字段与机器错误码使用英文。
- 注释重点解释设计原因、约束和非显然取舍，不逐句复述代码。

## 6. 技术与依赖基线

- Python 最低版本为 3.14。
- 使用 `pyproject.toml` 管理项目和依赖，使用 `uv.lock` 锁定版本。
- 不同时维护手工编辑的 `requirements.txt` 等第二套依赖清单；如工具需要导出，必须注明生成方式与用途。
- 添加依赖前确认标准库或现有依赖无法合理完成任务，并说明引入理由。
- 新依赖必须检查 Python 3.14 兼容性、许可证和维护状态。
- 修改依赖后必须更新 `uv.lock` 并验证锁定安装。
- 生产数据库为 PostgreSQL；开发和自动化测试可以使用 SQLite，但业务代码不得依赖 SQLite 特性。
- SQLAlchemy 使用 2.x 风格，数据库结构只允许通过 Alembic 迁移修改。
- 禁止以应用启动时的 `create_all()` 代替正式迁移。

### 6.1 生产部署延后

- 根据 ADR-0009，首期全部功能完成并通过功能验收前，不设计、创建或验收生产部署。
- 当前不创建生产 Caddyfile、生产 Docker Compose 拓扑、公网 DNS、证书、端口映射、Tailscale 配置、云主机发布流程或生产备份任务。
- 根据 ADR-0010，访问方向已冻结为受限公网，NiceGUI Web/BFF 是唯一公网应用入口，API、Worker、Redis、PostgreSQL 不得单独暴露公网；反向代理产品、生产拓扑、域名证书、生产密钥托管、备份和发布实现仍延后确认。
- 允许为本地开发和自动化测试提供 PostgreSQL、Redis 等最小依赖启动方式，但不得将其声称或扩展为生产拓扑。
- 真实浏览器本地登录调试可在显式开发配置下仅将 Cookie `Secure` 设为 `false`，但进程只能绑定回环地址；非回环地址与该配置组合时必须拒绝启动。其他 Cookie、CSRF 和授权要求不放宽。

## 7. 服务边界与依赖方向

项目采用 monorepo，但 NiceGUI Web、FastAPI API 和后台 Worker 是独立运行单元。这一边界不代表当前已确认生产部署拓扑。

### 7.1 NiceGUI Web

- 负责页面展示、交互和用户反馈。
- 只能通过 FastAPI API 使用业务能力。
- 禁止导入 ORM 模型、Repository 或直接连接数据库。
- 不在页面事件处理器中实现权限、事务或核心业务规则。

### 7.2 FastAPI API

- 负责认证、授权、输入校验、事务边界和业务编排。
- 所有权限必须在服务端校验，不能只隐藏页面按钮。
- API 内部保持简单；不要为每个类机械创建接口、工厂或依赖容器。

### 7.3 后台 Worker

- AI 生成和其他长任务通过 Redis 队列异步执行。
- API 创建任务后返回任务 ID；Web 使用事件推送或轮询展示状态。
- 任务契约必须明确、可重试且具备幂等性。
- 重试不得造成重复版本、重复归档或成功内容被覆盖。

### 7.4 共享契约

- `packages/contracts` 只存放稳定的请求模型、响应模型、事件模型和枚举。
- 禁止在共享契约包中放置 ORM、Repository、业务逻辑或供应商客户端。
- Web 不得通过共享包绕过 API 服务边界。

### 7.5 外部集成

- AI、节假日、Word 导出和未来对象存储通过明确的集成模块或适配器访问。
- 业务服务不得直接散落供应商 SDK 或原始 HTTP 调用。
- 当前一日活动计划不接入天气服务，也不设置天气标签。
- 不提前实现照片、视觉识别或对象存储服务，也不创建对象存储接口、适配器、配置、表、页面或服务。仅在未来扩展边界中说明，照片能力获得明确授权前必须先设计私有存储、授权、短时签名地址、审计、保留期限和彻底删除。

## 8. 数据模型与园所隔离

当前为单园部署，但必须为未来多园扩展保留隔离边界。

- 所有园所范围业务表必须包含 `kindergarten_id`、`created_at`、`updated_at`。
- 可变业务记录按需要增加 `created_by`、`updated_by`，不使用含义模糊的通用 `user_id` 代替明确审计字段。
- Repository 查询必须显式带上 `kindergarten_id`；禁止先读取跨园数据再在业务层过滤。
- 写入、更新、删除和唯一性校验同样必须受 `kindergarten_id` 约束。
- 全局枚举、迁移元数据等非园所数据可以例外，但应在模型附近说明原因。
- 涉及数据模型的修改必须包含迁移、模型、Repository 和相应测试。

## 9. 首期必要设置

不得以“未来子系统需要”为理由提前实现无业务依赖的设置页面。首期只提前上线一日活动计划直接依赖的设置：

- 幼儿园信息：名称。数据库保留 `timezone` 字段，首期固定为 `Asia/Shanghai`；管理员只能查看，当前园所设置只允许修改园所名称。真正支持其他时区时再通过迁移、日期规则设计和新验收开放编辑。
- 学期管理：学期名称、起止日期、当前学期。
- 用户与班级：教师账号、班级、年龄段、教师关联和主班教师。
- 班级区域：由任课教师维护本班室内与户外区域。
- AI 模型档案：API 地址、加密密钥、模型名和能力标签。
- AI 提示词管理：草稿、发布、历史版本和回滚。

后续游戏观察记录、一对一倾听、照片上传、视觉识别和对象存储必须等到明确任务下达后再实现，不创建空页面、空表或未使用的部署服务。

## 10. 一日活动计划业务不变量

以下规则不得为了实现方便而改变：

- 同一 `(kindergarten_id, class_id, plan_date)` 只能存在一份当前教案，并由数据库唯一约束保证。
- 学期开始日期所在自然周为第一周；此后每到下一个周一，教学周次递增一周。
- 所选日期不在学期内、不是工作日或无法确认工作日时，只显示软提示，不阻止保存。
- 教案不增加草稿、完成、审批等状态，只以 `archived_at` 表示归档。
- 归档后默认只读；恢复归档后才能继续编辑。
- 自动保存只更新当前内容，不创建历史快照。
- 手动保存、归档、恢复归档、恢复历史版本及教师采用 AI 预览时创建历史快照；AI 生成、重试、拒绝和失败本身不创建快照。
- 页面同时支持显式保存和停止输入约 3 秒后的自动保存，并显示保存状态。
- 并发修改使用版本号进行乐观锁校验，禁止最后写入者静默覆盖他人内容。

教师与班级采用多对多关系。教师只能访问本人关联班级的教案，管理员可以访问全园教案；权限判断必须由 API 执行。

## 11. AI 与提示词规则

### 11.1 模型访问

- 使用统一的 OpenAI 兼容接口，不硬编码具体供应商。
- 管理员可以配置多个模型档案，并声明 `text`、`vision`、`structured_output` 等能力。
- 业务任务只能选择满足提示词能力要求的模型档案。
- 常规测试禁止调用真实或付费 AI 接口。

### 11.2 提示词管理

- 禁止在业务代码中散落可运营提示词。
- 提示词通过提示词管理子系统按稳定标识和已发布版本读取。
- 每个提示词定义输入变量、所需能力和模型档案选择。
- 占位符只允许白名单纯替换词法 `\{\{[ \t]*([a-z][a-z0-9_]*)[ \t]*\}\}`；不得执行表达式、过滤器、循环、函数、嵌套字段访问或递归渲染变量值，数组与对象使用稳定 JSON 序列化。
- 业务生成只能使用已发布版本。
- 提示词修改、发布和回滚仅允许管理员执行并记录审计事件。
- 每次生成记录提示词版本和模型信息，但不记录 API Key。

### 11.3 生成行为

- 一键生成只生成晨间活动、晨间谈话、室内区域游戏和下午户外游戏四栏；不生成集体活动或一日活动反思。
- 一日活动反思只能在前五个上游栏目满足结构化完整性后，由教师显式发起生成。
- 集体活动必须先由教师粘贴文本或上传 `.docx` 原始教案；拆分/补全与新增适龄环节分别生成、预览和采用。教师先采用并保存拆分结果后，才能生成新增环节；新增失败不得回滚已采用的拆分结果。
- 每个栏目必须支持独立生成和重新生成。
- 一键生成的栏目作为独立子任务执行；部分失败不得回滚成功栏目。
- AI 结果必须经过结构化模型校验，失败时最多自动重试两次。
- 生成时显示“AI 正在生成，请稍候”；重试时显示“生成失败，正在重试”。
- 最终失败后保留原内容，并允许只重试失败栏目。
- AI 不得静默覆盖教师已经编辑的内容。
- AI 生成、重新生成、重试、拒绝和失败只记录任务、预览结果与审计事件；只有教师明确采用仍有效的 AI 预览时才更新正文并创建关键版本快照。

## 12. Word 模板与导出

- 未经明确任务授权，不得重建、重新排版或覆盖原始 `.docx` 模板。
- 导出必须基于模板副本，运行时文件只能写入专用导出目录或临时目录。
- 严格保持模板表格结构、字体、字号、段落和换行要求。
- 仅 AI 新增的集体活动环节使用红色字体；其他 AI 处理内容不得整体标红。
- 幼儿园名称和编写教师从业务数据读取，不硬编码进导出逻辑。
- 修改导出逻辑后必须验证字段位置、样式、换行和原模板哈希或文件状态未改变。
- 测试和示例导出不得包含真实教师或幼儿身份信息。

## 13. 安全与隐私硬性规则

- WebAuthn 通行密钥是唯一常规登录方式；不得新增密码、OTP、微信小程序 MFA 或默认管理员作为隐藏兼容或恢复入口。
- 不开放公众注册。空系统只允许部署控制台一次性初始化待激活首位管理员；其他账号由管理员创建并发放单次邀请，邀请完成后仍须管理员带外核验激活。
- 紧急恢复必须同时使用离线恢复码和人工核验；最后管理员恢复还必须由预登记园所负责人及独立运维/安全责任人双人核验，不得设置万能恢复码或数据库改值后门。
- 禁止提交 `.env`、真实 API Key、JWT 密钥、数据库口令、备份、运行时导出教案和幼儿照片。
- AI Key 使用 AES-256-GCM 加密保存；每次加密生成随机 96 位 nonce，AAD 绑定园所 ID、模型档案 ID 和 envelope version。认证标签验证失败必须拒绝解密，不得返回部分明文。通过 `key_id` 支持新密钥写入、旧密钥只读、分批重加密和验证后移除。
- 主加密密钥存放在数据库之外；开发环境从仓库外、受文件权限保护的本地密钥文件读取，自动化测试使用明确的测试适配器。主密钥不得进入代码库、数据库、镜像、普通环境变量、日志、异常、审计或测试快照。生产密钥托管方案延后至 M9。页面只显示脱敏值。
- 日志、异常、任务消息、测试快照和审计记录不得包含密钥、令牌、完整敏感教案或照片内容。
- 外部 AI 请求遵循数据最小化原则，只发送当前任务必需内容，默认排除教师账号和幼儿身份信息。
- 管理员首次启用模型档案时必须确认外部服务的数据处理风险，并记录启用人和时间。

- 文本/内容搜索：`rg`（ripgrep）。

### 已安装工具与技能

### AST-grep 使用

- 在运行复杂模式前，需说明意图并展示确切的命令。
- 常见查询：
  - 查找从 `node:path` 的导入（TypeScript/TSX）：
    - `ast-grep -p "import $ from 'node:path'" src --lang ts,tsx,mts,cts`
  - 查找 CommonJS 的 `node:path` 的 require：
    - `ast-grep -p "require('node:path')" src --lang js,cjs,mjs,ts,tsx`
  - 建议重写（除非获批，否则不要在代码中自动应用）：
    - 搜索：`ast-grep -p "import $ from 'node:path'" src --lang ts,tsx`
    - 建议的替换：`import $ from 'pathe'`

### 搜索规范（fdfind/rg/sg）

- 排除大文件夹以保持搜索快速和相关性：`.git`、`node_modules`、`coverage`、`out`、`dist`。
- 优先对特定路径（如 `src`）进行搜索，以隐式避开供应商和版本控制目录。
- 示例：
  - `rg -n "pattern" -g "!{.git,node_modules,coverage,out,dist}" src`
  - `fdfind --hidden --exclude .git --exclude node_modules --exclude coverage --exclude out --exclude dist --type f ".tsx?$" src`
- ast-grep 通常遵守 `.gitignore`；指定 `src` 以避免扫描供应商文件夹：
  - `ast-grep -p "import $ from '@shared/$'" src --lang ts,tsx,mts,cts`
  - 如需添加忽略模式，请将其添加到您的忽略文件中，而非禁用忽略功能。
