## 3. 项目定位

Child Manager 是面向幼儿园日常教育工作的 Cloud 教育管理系统。

首期目标是交付可在线使用的一日活动计划管理系统，完成以下闭环：

```text
必要设置
  → 教案创建与自动保存
  → 分栏目 AI 生成
  → 教师确认与编辑
  → 归档与历史版本
  → 固定 Word 模板导出
```

系统当前服务于单所幼儿园，但所有园所业务数据保留 `kindergarten_id`，为未来多园隔离预留边界。首期不是多园 SaaS 产品，也不提供多园运营界面。

后续计划接入游戏观察记录、一对一倾听和视觉识别等子系统，但首期只实现这些子系统共同依赖的基础能力，不提前实现其业务页面和数据表。

## 5. 不可误读的已确认决策

以下是快速摘要。详细规则以 README、AGENTS 和模板说明为准。

### 5.1 产品与部署

- 只交付 Cloud 版本，不再开发本地系统。
- 首期单园部署，核心业务表保留 `kindergarten_id`。
- NiceGUI Web 与 FastAPI API 从首期开始独立部署。
- PostgreSQL 是生产数据库；SQLite 仅用于开发和自动化测试。
- Redis 与独立 Worker 执行 AI 等长任务。
- 后台任务采用 Dramatiq 2.x + Redis，PostgreSQL 中的任务记录是权威状态。
- 任务进度首期使用 1–2 秒短轮询。
- 生产部署、访问网络、反向代理和密钥托管实现延后到全部功能完成后复审；功能开发期间不实施公网或 Tailscale 部署。
- 对象存储只预留边界，首期不部署。
- 界面仅提供简体中文，业务时区为 `Asia/Shanghai`，数据库时间戳存储为 UTC。数据库保留 `timezone` 字段，首期固定为 `Asia/Shanghai`；管理员只能查看，当前园所设置只允许修改园所名称。真正支持其他时区时再通过迁移、日期规则设计和新验收开放编辑。

### 5.2 用户与权限

- 首期只有管理员和教师两类角色。
- 不开放公众注册；首位管理员初始化后，由管理员创建教师账号。
- 管理员管理全园设置并访问全部教案。
- 教师只能访问本人关联班级的教案，并维护本人任教班级的室内、户外区域。
- 教师与班级为多对多关系，班级可以设置主班教师。
- 所有权限由 API 服务端校验，不能只依赖 Web 隐藏入口。

### 5.3 数据与架构边界

- Web 只能调用 API，不得直接连接数据库或导入 ORM、Repository。
- API 负责认证授权、事务边界、业务编排和园所隔离。
- Worker 通过明确、可重试、幂等的任务契约工作。
- `packages/contracts` 只保存稳定请求、响应、事件模型和枚举。
- 所有园所业务查询、写入和唯一性校验必须显式带上 `kindergarten_id`。
- 数据库结构只能通过 Alembic 修改，禁止用 `create_all()` 代替迁移。

### 5.4 一日活动计划

- 同一 `(kindergarten_id, class_id, plan_date)` 只能存在一份当前教案。
- 学期开始日期所在自然周为第一周；此后每到下一个周一，教学周次递增一周。
- 非工作日、学期外日期和无法确认工作日都只做软提示，不阻止保存。
- 教案不设置草稿、完成、审批等状态，只使用 `archived_at`。
- 归档后默认只读，恢复后才能继续编辑。
- 同时支持显式保存与停止输入约 3 秒后的自动保存。
- 自动保存不创建历史快照；手动保存、归档、恢复归档、恢复历史版本及教师采用 AI 预览时创建快照。AI 生成、重试、拒绝和失败只记录任务、结果和审计。
- 使用版本号进行乐观锁校验，禁止静默覆盖并发修改。

### 5.5 AI 与提示词

- 使用 OpenAI 兼容接口，不硬编码供应商。
- 管理员可配置多个模型档案以及 `text`、`vision`、`structured_output` 等能力。
- 提示词管理是独立系统能力，支持草稿、发布、历史版本和回滚。
- 提示词占位符仅使用白名单纯替换词法 `\{\{[ \t]*([a-z][a-z0-9_]*)[ \t]*\}\}`；不执行表达式、过滤器、循环、函数或嵌套字段访问。
- 业务生成只能使用已发布提示词版本。
- 一键生成仅包含晨间活动、晨间谈话、室内区域游戏和下午户外游戏；反思在前五个上游栏目完整后由教师显式发起，集体活动不在一键范围内。
- 每个栏目必须支持独立生成和重新生成。集体活动必须先由教师粘贴文本或上传 `.docx` 原始教案，拆分/补全与新增环节分别生成、预览和采用。
- 各栏目独立执行；部分失败不能回滚成功栏目。
- 结构校验失败时最多自动重试两次。
- 生成时显示“AI 正在生成，请稍候”，重试时显示“生成失败，正在重试”。
- AI 不得静默覆盖教师内容，外部请求必须遵循数据最小化原则。

### 5.6 Word 导出

- 首期只导出 `.docx`，不导出 PDF。
- 原始模板不得被覆盖、重排或从零替代。
- 导出必须基于模板副本并保持表格、字体、字号、段落和换行。
- 只有 AI 新增的集体活动环节使用红色字体。
- 幼儿园名称与编写教师从业务数据读取，不得硬编码。

## 6. 首期必要设置

设置中心是教案闭环的前置依赖，必须在 AI 教案功能前可用：

1. 幼儿园信息：名称、默认时区。
2. 学期管理：名称、起止日期、当前学期。
3. 用户与班级：教师账号、班级、年龄段、教师关联、主班教师。
4. 班级区域：任课教师维护本班室内与户外区域。
5. AI 模型档案：API 地址、加密密钥、模型名、能力标签。
6. AI 提示词管理：草稿、发布、历史版本、回滚。

只实现首期直接依赖的设置。不得提前创建照片、观察记录或其他未来子系统的空设置页面。

## 7. 共同实施路线

`codex` 与 `trae` 必须遵循相同的里程碑依赖顺序，但允许内部实现细节不同。不得为了更快展示页面而跳过认证、设置、迁移或服务边界。

### 阶段 1：工程骨架

- 建立 monorepo 结构、NiceGUI Web、FastAPI API、共享契约和 Worker 入口。
- 建立 `pyproject.toml`、`uv.lock`、Ruff、Pyright、Pytest 和 GitHub Actions。
- 为本地开发和自动化测试提供 PostgreSQL、Redis 等最小依赖与健康检查，不建设生产拓扑。
- Codex 与 Trae 使用独立 worktree、Compose 项目、回环端口、数据库和运行时目录，证明两套环境可同机并行且互不影响。

### 阶段 2：认证授权

- 首位管理员安全初始化。
- 管理员创建教师账号。
- Argon2id 密码、访问令牌、刷新令牌和服务端权限校验。

### 阶段 3：必要设置

- 实现幼儿园、学期、教师、班级、年龄段和班级区域设置。
- 完成 `kindergarten_id` 数据隔离与迁移测试。

### 阶段 4：AI 基础能力

- 实现多个模型档案、密钥加密和能力标签。
- 实现提示词草稿、发布、版本和回滚。

### 阶段 5：教案手工闭环

- 实现日历与列表视图、日期规则、唯一约束、编辑、自动保存、乐观锁、归档和历史快照。
- 在没有 AI 的情况下先完成可验证的手工教案闭环。

### 阶段 6：AI 异步生成

- 接入 Redis Worker、任务状态、栏目级生成、结构校验、重试和失败恢复。
- 支持集体活动文本与 `.docx` 导入。

### 阶段 7：Word 导出

- 按固定模板填充完整教案。
- 验证字段位置、中文字体、换行和新增环节红字。

### 阶段 8：功能验收

- 完成首期功能、权限、数据隔离、审计、日志和迁移验收。
- 完成桌面浏览器与平板关键流程验收。

### 阶段 9：生产部署复审（功能完成后）

- 执行威胁建模，决定私有网络或其他访问入口。
- 另行冻结生产拓扑、密钥托管、备份恢复与上线流程，不属于当前功能开发阶段。

## 8. 当前非目标

首期明确不做：

- 本地版、桌面版、离线模式或桌面安装包。
- 多园 SaaS 管理界面。
- 教案提交、审核、园长审批等工作流。
- 天气服务或天气标签。
- PDF、图片 OCR 教案导入。
- PDF 导出和在线打印排版。
- 全文搜索。
- 手机端完整教案排版编辑。
- 游戏观察、一对一倾听、照片上传和视觉识别业务。
- S3、MinIO 或其他对象存储的实际部署。
- 从旧仓库迁移 MySQL、本地 SQLite 产品模式或 NiceGUI 一体化架构。
- 为未来需求提前创建未使用的表、页面、服务或抽象。

## 11. 高风险点

| 风险 | 可能后果 | 既定缓解措施 |
| --- | --- | --- |
| AI 结构化输出波动 | 字段缺失、格式错误、错误覆盖 | Schema 校验、栏目级任务、最多两次重试、保留原内容 |
| Word 模板结构被改变 | 导出错位、字体或红字错误 | 锁定模板、基于副本导出、结构与样式回归测试 |
| Redis 任务重复执行 | 重复版本、重复写入、内容覆盖 | 幂等键、任务状态、数据库约束和版本校验 |
| SQLite 与 PostgreSQL 行为差异 | 测试通过但生产失败 | 避免 SQLite 特性，执行 PostgreSQL 迁移与关键集成测试 |
| 节假日服务不可用 | 日期判断失败或阻塞教案 | 本地库优先、在线补充、缓存、软提示降级 |
| 跨园数据泄露 | 严重隐私与权限事故 | Repository 强制 `kindergarten_id`，API 服务端鉴权与隔离测试 |
| AI 覆盖人工内容 | 教师修改丢失 | 教师确认、乐观锁、历史快照、失败不回滚成功栏目 |
| 两个实现分支文档漂移 | Agent 依据过期规则开发 | 共享文档独立提交、cherry-pick、分支状态与最后验证记录 |
| 密钥或幼儿数据进入日志 | 敏感信息泄露 | 加密、脱敏、数据最小化、结构化日志字段审查 |

| 未来功能提前实现 | 首期延期、架构空壳增加 | 严格非目标、按阶段实施、只预留边界不落地服务 |

## 12. 系统架构基线

服务边界、Dramatiq 任务可靠性、短轮询和认证安全结果已经确认，统一见 [`docs/design/system-architecture.md`](docs/design/system-architecture.md)。根据 ADR-0009，生产部署、访问网络、反向代理和密钥托管实现已延后，不得在功能完成前固化。

数据表与 API 契约、Cookie/CSRF 参数、任务租约、主密钥轮换、备份工具和生产资源限制仍需在相应详细设计或实现任务中细化，但不能改变已确认的安全与用户行为。

## 13. 验证基线

实现分支的标准质量命令为：

```bash
uv sync --locked
uv run ruff format --check .
uv run ruff check .
uv run pyright
uv run pytest
```

当前 `main` 没有 Python 项目骨架，因此这些命令尚不可执行。不得把“尚无命令”误写为“验证通过”。

各阶段还必须执行与风险匹配的专项验证：

- 数据模型：Alembic 升级与园所隔离测试。
- API：认证、权限、输入校验和错误响应。
- Worker：重试、幂等、部分失败和状态反馈。
- Word：模板未改变、字段位置、字体、换行和红字范围。
- Docker：服务健康检查、依赖就绪和生产配置检查。

## 14. CONTEXT 更新规则

出现以下情况时必须更新本文档：

- 产品范围或架构决策发生变化。
- 完成一个实施阶段或开始下一阶段。
- 数据库迁移、部署拓扑或关键接口发生变化。
- 出现会影响后续 Agent 的已知问题、阻塞或临时方案。
- 标准命令、测试基线或最后验证发生变化。
- 当前分支状态发生实质变化。

每次更新：

- 更新文首日期与当前上下文分支。
- 只修改当前分支对应的状态。
- 记录可复现的最后验证，不写模糊的“应该可用”。
- 保持摘要性质，将详细设计写入 `docs/` 并在此链接。
- 删除已失效的信息，不无限追加历史流水账。

如果上下文与代码不一致，应先查明原因，再修正文档或实现；不得为了让文档“看起来正确”而掩盖实际状态。
